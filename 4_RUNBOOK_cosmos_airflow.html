CD

Airflow is your production deployment mechanism

Before touching a single DAG file, understand what role Airflow plays in the four-environment pipeline.

CI/CD — two different things. CI (Continuous Integration) — automated testing on every Pull Request. GitHub Actions runs dbt build --select state:modified+ to check your code before it can merge. Covered in Runbook 5.

CD (Continuous Deployment) — automated deployment to production. Airflow runs dbt build --target prod on a schedule and whenever code merges to main. This is what you are setting up in this runbook.
The golden rule: nobody manually deploys to production. You will never type dbt run --target prod yourself. That command only runs inside Airflow — either on a schedule or triggered automatically when a Pull Request merges to the main branch.

Every change to production has a paper trail: a Pull Request, a code review, a CI check, and an Airflow run log. If something goes wrong, you can trace exactly what changed, when, and who approved it.
Airflow only ever touches production schemas. Full stop. Airflow runs dbt build --target prod. This writes to prod_staging_silver and prod_mart_gold and nowhere else. Airflow has no knowledge of your personal schema or dev schemas — it does not touch them, read from them, or overwrite them.

Core concepts

ConceptWhat it isPlain-English analogy
DAGA Python file defining a complete workflow — its tasks, dependencies, and scheduleA recipe — lists every step in order and what to do if a step fails
TaskA single unit of work inside a DAGOne line in the recipe — "fold in the eggs"
DAG RunOne execution of a DAG — triggered manually or by scheduleOne time you actually cook the recipe
ConnectionA saved set of credentials Airflow uses to talk to PostgreSQLThe phone number saved in your contacts
ScheduleA cron expression that tells Airflow when to trigger a DAG automaticallyThe alarm clock — set it once, it fires every time
What is a cron expression? Cron uses a 5-field expression to define when something should run:

* * * * * — minute · hour · day-of-month · month · day-of-week

Examples:
0 */6 * * * — every 6 hours (00:00, 06:00, 12:00, 18:00)
0 2 * * * — every day at 2:00am
None — never runs automatically — manual trigger only

Why Cosmos?

Without Cosmos, running dbt from Airflow means writing a BashOperator task that executes dbt run as a shell command. That works — but it's a black box. Airflow sees one task: "run dbt." If 8 of your 12 staging models succeed and 4 fail, Airflow just reports the whole thing as failed.

Cosmos reads your dbt project — the models, the tests, the dependencies between them — and converts each dbt model into its own individual Airflow task. Instead of one opaque task, you get a full task graph where every model is visible, every test is trackable, and a failure in one model doesn't hide which model caused it.

dbt project
models, tests, sources
→
Cosmos
DbtTaskGroup
→
Airflow DAG
one task per model
→
PostgreSQL
tables built & tested
How does Cosmos know the model order? dbt generates manifest.json every time you run dbt compile or dbt build. This file contains the full dependency graph for every model in the project. Cosmos reads this manifest and wires up the Airflow task dependencies automatically.

Key principle: dbt owns the order, you own the schedule. You never define staging-before-marts in Airflow code — dbt's lineage graph already knows that.

The automated pipeline

This runbook sets up one DAG that runs the full pipeline on a daily schedule. Cosmos reads dbt's manifest and builds staging → intermediate → marts in the correct order automatically.

DAG schedule
pipeline_daily Every day at 2:00am — full pipeline: staging → intermediate → marts
Relationship tests and first run. Do not add relationships tests to mart-layer schema.yml files until both the staging and mart tables have been built and exist in the database. The test will fail with a "relation does not exist" error on the first pipeline run. Add relationship tests only after the full pipeline has run successfully end to end.
1

Create production schemas

dbt does not create schemas automatically — only tables inside them. Run this once before triggering any DAG:

Why weren't these created in Runbook 2? Runbook 2 created your personal and dev schemas. Production schemas are deliberately kept separate — they are owned by the automated pipeline, not by individual developers. Creating them here, immediately before setting up the pipeline that writes to them, keeps the separation clear. 
docker exec postgres psql -U mds_user -d btg_resource_utilization -c "
CREATE SCHEMA IF NOT EXISTS prod_staging_silver;
CREATE SCHEMA IF NOT EXISTS prod_mart_gold;
CREATE SCHEMA IF NOT EXISTS prod_snapshots;
GRANT ALL ON SCHEMA prod_staging_silver TO mds_user;
GRANT ALL ON SCHEMA prod_mart_gold TO mds_user;
GRANT ALL ON SCHEMA prod_snapshots TO mds_user;
GRANT USAGE ON SCHEMA prod_staging_silver TO analytics_engineer, data_scientist;
GRANT USAGE ON SCHEMA prod_mart_gold TO analytics_engineer, business_user;
SELECT schema_name FROM information_schema.schemata
WHERE schema_name LIKE 'prod%' ORDER BY schema_name;"
Three rows returned: prod_mart_gold, prod_snapshots, prod_staging_silver.

Refresh pgAdmin — you should now see all three prod schemas under btg_resource_utilization → Schemas.

2

Create the Airflow connection

Cosmos connects to PostgreSQL through an Airflow connection — a named set of credentials stored in the Airflow metadata database. The DAG file references this connection by name (btg_resource_utilization) rather than hardcoding credentials.

Create it via terminal — more reliable than the UI and repeatable by anyone who clones the repo:

docker exec btg-airflow-scheduler airflow connections add btg_resource_utilization \
    --conn-type postgres \
    --conn-host postgres \
    --conn-login mds_user \
    --conn-password mds_password \
    --conn-port 5432

Verify it was created:

docker exec btg-airflow-scheduler airflow connections get btg_resource_utilization
You see the connection details with conn_id=btg_resource_utilization and conn_type=postgres.
Why is the host postgres and not localhost? Airflow and PostgreSQL both run inside Docker on the same network. Inside Docker, containers find each other by container name — not by localhost. The PostgreSQL container is named postgres in docker-compose.yml. From outside Docker (your Mac Terminal or pgAdmin), you use localhost:5432. From inside Docker (Airflow), you use postgres:5432.
How do you verify the connection works? Airflow 2.x removed the connections test CLI command. The correct way to verify is to trigger the DAG in Step 5 and check its logs — if the connection is wrong, the first task will fail immediately with a clear "could not connect" error. Fix it in Admin → Connections and re-trigger.

Why we're not using Cosmos (yet)

Cosmos is installed in your Docker image. This section explains what it does, why it's appealing, and why plain dbt build is the better choice at this stage — so you can make an informed decision when your project grows.

What Cosmos gives you. Without Cosmos, Airflow sees your entire pipeline as one task: "run dbt." If 8 of 12 models succeed and 4 fail, Airflow just reports the whole thing as failed. You have to dig through logs to find out what went wrong.

Cosmos reads your dbt project and converts each model into its own Airflow task. You get a full task graph, model-level retry, and task-level run history. At large scale (100+ models, long-running pipelines), this visibility is genuinely valuable.
What Cosmos takes away. At small scale, Cosmos introduces problems that plain dbt build doesn't have:

1. Test ordering breaks. dbt runs tests after each model in dependency order. Cosmos runs tests per-task independently — so a staging model's test can fire before the mart models it references are built. Relationship tests fail with "relation does not exist" on every first run.

2. Manifest staleness. Cosmos builds its Airflow task graph from manifest.json. If you add a new model or change a test without recompiling, Cosmos builds the wrong graph — new models get orphaned, changed tests don't run. You need a dbt compile step before every pipeline run.

3. Schema double-append. Cosmos overrides the dbt target schema via profile_args. If you set it to prod_staging_silver, dbt appends again and produces prod_staging_silver_staging_silver. It must be set to just prod and requires understanding of how dbt combines target + custom schema.

4. More moving parts. The Airflow connection, ProfileConfig, ExecutionConfig, DbtTaskGroup, and RenderConfig all need to be wired correctly. Any misconfiguration is silent — the DAG loads fine but runs wrong.
When to switch to Cosmos. Consider it when: your pipeline takes longer than 10 minutes and you need to retry individual failed models without rerunning everything, or you have a large team where different people own different models and need model-level visibility to debug independently.

For now, plain dbt build in a BashOperator is the right choice — dbt handles ordering correctly, tests run at the right time, and there's nothing to misconfigure.

3

Add the DAG file

Why plain dbt build instead of Cosmos? Cosmos converts each dbt model into its own Airflow task — great for visibility, but it breaks dbt's natural test ordering. With Cosmos, a staging model's tests can fire before the mart models it references are built, causing "relation does not exist" errors on relationship tests.

Plain dbt build lets dbt handle the order internally. Staging builds first, intermediate next, marts last — and tests run after each model in the correct sequence. Simpler, more robust, and correct for a project of this size.

Remove any old DAG files (silver_layer_staging.py, silver_layer_staging_daily.py, gold_layer_marts_daily.py, pipeline_6hours.py) then create the new one:

code airflow/dags/pipeline_daily.py

Paste this exactly, then save with ⌘ + S:

from datetime import datetime
from airflow import DAG
from airflow.operators.empty import EmptyOperator
from airflow.operators.bash import BashOperator

DBT_PROJECT_PATH = "/opt/airflow/dbt"
DBT_EXECUTABLE   = "/usr/local/airflow/dbt_venv/bin/dbt"

with DAG(
    dag_id="pipeline_daily",
    start_date=datetime(2026, 1, 1),
    schedule="0 2 * * *",   # every day at 2:00am
    catchup=False,
    tags=["daily", "full"],
) as dag:

    start = EmptyOperator(task_id="start")
    end   = EmptyOperator(task_id="end")

    run_pipeline = BashOperator(
        task_id="dbt_build",
        bash_command=(
            f"cd {DBT_PROJECT_PATH} && "
            f"{DBT_EXECUTABLE} build "
            f"--target prod "
            f"--profiles-dir {DBT_PROJECT_PATH}"
        ),
    )

    start >> run_pipeline >> end
Use schedule= not schedule_interval=. schedule_interval is deprecated in Airflow 2.4+ and will be removed in Airflow 3. Always use schedule=.

Syntax check before Airflow picks it up

Run the file directly inside the container — catches syntax errors immediately without waiting for Airflow's 30-second scan cycle:

docker exec btg-airflow-scheduler python /opt/airflow/dags/pipeline_daily.py
No errors and no traceback. Just the standard Airflow startup warnings — those are normal, ignore them.
4

Verify DAG appears in Airflow

Airflow's scheduler watches the dags/ folder and picks up new files automatically. Wait 30 seconds, then check:

docker exec btg-airflow-scheduler airflow dags list | grep pipeline
You see pipeline_daily listed. It shows a pause icon in the Airflow UI at localhost:8080 — new DAGs are paused by default.

If the DAG does not appear after 60 seconds, force a reserialize:

docker exec btg-airflow-scheduler airflow dags reserialize
Why are new DAGs paused by default? Airflow pauses every new DAG automatically so it does not start running before you have reviewed it. You must explicitly unpause a DAG to enable its schedule — covered in Step 6.
5

Trigger manually and verify

Before enabling the schedule, trigger the pipeline manually to confirm everything runs end to end.

docker exec btg-airflow-scheduler airflow dags trigger pipeline_daily

The DAG run is queued. You can watch it in the Airflow UI at localhost:8080 — you will see a single dbt_build task. Or verify entirely from the terminal:

Check run status from the terminal

# Wait ~30 seconds then check — look for state = success
docker exec btg-airflow-scheduler airflow dags list-runs --dag-id pipeline_daily
The most recent run shows state = success. The run should complete in under 60 seconds.

Verify data actually landed in prod

A success state in Airflow confirms dbt ran — but this confirms the data is actually in the database:

docker exec postgres psql -U mds_user -d btg_resource_utilization -c "
SELECT 'prod_staging_silver' AS schema, COUNT(*) FROM prod_staging_silver.stg_customer_details
UNION ALL
SELECT 'prod_mart_gold', COUNT(*) FROM prod_mart_gold.model_revenue_monthly;"
Both rows return a count greater than 0. This confirms data flowed from bronze → staging → marts in prod.

If the run failed — find out why

# Step 1 — get the run_id from the failed run
docker exec btg-airflow-scheduler airflow dags list-runs --dag-id pipeline_daily

# Step 2 — check task states for that run_id
docker exec btg-airflow-scheduler airflow tasks states-for-dag-run pipeline_daily <run_id>

Then open the Airflow UI → click the failed run → click dbt_build task → click Log. The full dbt output is there — look for lines starting with FAIL or ERROR.

Common failure causes:

— Connection error: confirm the Airflow connection host is postgres not localhost
— profiles.yml not found: confirm --profiles-dir points to /opt/airflow/dbt
— dbt binary not found: confirm path is /usr/local/airflow/dbt_venv/bin/dbt
The full pipeline ran successfully. Bronze → staging → intermediate → marts — all run by dbt in the correct order, orchestrated by Airflow on a schedule.
6

Enable the schedule

The manual run succeeded. Enable the schedule so the pipeline runs automatically going forward.

docker exec btg-airflow-scheduler airflow dags unpause pipeline_daily
The DAG shows a green active indicator in the Airflow UI. The Next Run column shows tomorrow at 2:00am.
docker exec btg-airflow-scheduler airflow dags next-execution pipeline_daily
Returns a future timestamp — the next 2:00am UTC.
What happens if the stack is not running at 2:00am? Because catchup=False is set, Airflow runs one catch-up run at the next scheduled interval once the stack comes back up — not a backlog of every missed run.

Spot-check the data

Run a spot-check query on the gold layer to confirm data is fresh:

docker exec postgres psql -U mds_user -d btg_resource_utilization -c "
SELECT
  model_variant,
  month_start_date,
  total_net_revenue
FROM prod_mart_gold.model_revenue_monthly
ORDER BY month_start_date DESC, total_net_revenue DESC

5 rows returned with model names, month dates, and revenue figures.

Troubleshooting

DAG does not appear in Airflow after 60 seconds ▼
# Check for import errors
docker exec btg-airflow-scheduler airflow dags list-import-errors

# Syntax check the file directly
docker exec btg-airflow-scheduler python /opt/airflow/dags/pipeline_daily.py

# Force reserialize if DAG still won't appear
docker exec btg-airflow-scheduler airflow dags reserialize

# Restart scheduler as last resort
docker compose restart airflow-scheduler
How to check if the last run succeeded (no UI) ▼
docker exec btg-airflow-scheduler airflow dags list-runs --dag-id pipeline_daily

Look for state = success on the most recent run. Then confirm data actually landed:

docker exec postgres psql -U mds_user -d btg_resource_utilization -c "
SELECT 'prod_staging_silver' AS schema, COUNT(*) FROM prod_staging_silver.stg_customer_details
UNION ALL
SELECT 'prod_mart_gold', COUNT(*) FROM prod_mart_gold.model_revenue_monthly;"

Both rows must return a count greater than 0. Airflow state alone confirms dbt ran — the row counts confirm data actually landed.

Run failed — how to find the error ▼
# Step 1 — get the run_id
docker exec btg-airflow-scheduler airflow dags list-runs --dag-id pipeline_daily

# Step 2 — check task states for that run_id
docker exec btg-airflow-scheduler airflow tasks states-for-dag-run pipeline_daily <run_id>

Then open Airflow UI → click the failed run → click dbt_build → click Log. The full dbt output is there. Look for lines starting with FAIL or ERROR — dbt prints the exact model name and SQL that failed.

Task fails with "could not connect to server" ▼

The profiles.yml prod target uses host: postgres — not localhost. From inside Docker, containers find each other by container name. Verify the profiles.yml inside the container:

docker exec btg-airflow-scheduler cat /opt/airflow/dbt/profiles.yml | grep host

All targets must show host: postgres. If any show localhost, fix dbt/profiles.yml on your Mac and restart the containers.

Task fails with "profiles.yml not found" ▼

The DAG uses --profiles-dir /opt/airflow/dbt. This path is the dbt folder mounted inside Docker via the volume in docker-compose.yml. Confirm the file is there:

docker exec btg-airflow-scheduler ls /opt/airflow/dbt/profiles.yml

If missing — the volume mount is broken. Check docker-compose.yml has ./dbt:/opt/airflow/dbt in both airflow service volumes sections.

Relationship test failing with "relation does not exist" ▼

A relationships test in a staging schema.yml references a mart table. This is a design issue — staging models should not reference downstream mart tables in their tests. Move the test to the mart model's schema.yml instead, where both tables are guaranteed to exist when the test runs.

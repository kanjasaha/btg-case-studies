CD

Airflow is your production deployment mechanism

Before touching a single DAG file, understand what role Airflow plays in the four-environment pipeline.

CI/CD — two different things. CI (Continuous Integration) — automated testing on every Pull Request. GitHub Actions runs dbt build --select state:modified+ to check your code before it can merge. Covered in Runbook 5.

CD (Continuous Deployment) — automated deployment to production. Airflow runs dbt build --target prod on a schedule and whenever code merges to main. This is what you are setting up in this runbook.
The golden rule: nobody manually deploys to production. You will never type dbt run --target prod yourself. That command only runs inside Airflow — either on a schedule or triggered automatically when a Pull Request merges to the main branch.

This is not just a convention. It is how data teams maintain trust in their production data. Every change to production has a paper trail: a Pull Request, a code review, a CI check, and an Airflow run log. If something goes wrong, you can trace exactly what changed, when, and who approved it.

How Airflow works

Apache Airflow is a pipeline scheduler. Its job is to run tasks in the right order, at the right time, and tell you when something goes wrong. Think of it as a school bell system for your data — it knows which class starts when, which room it's in, and sounds the alarm if a teacher doesn't show up.

Every pipeline in Airflow is defined as a DAG — a Directed Acyclic Graph. The name sounds technical but the idea is simple: a list of tasks where each task can depend on one or more tasks before it. Airflow reads the dependency graph, figures out the correct execution order, and runs everything accordingly.

What is a DAG? A DAG is just a Python file that describes a workflow. It defines:

Tasks — the individual units of work (run this dbt model, execute this SQL, send this alert)
Dependencies — which tasks must complete before others can start
Schedule — when the whole workflow should run (every 6 hours, every night at 1am, manually)

Airflow does not run your code directly — it orchestrates when and in what order your code runs.

The three core concepts you need to understand before looking at any DAG file:

ConceptWhat it isPlain-English analogy
DAG A Python file defining a complete workflow — its tasks, dependencies, and schedule A recipe — lists every step in order and what to do if a step fails
Task A single unit of work inside a DAG — run a dbt model, execute a shell command, call an API One line in the recipe — "fold in the eggs"
DAG Run One execution of a DAG — either triggered manually or by the schedule One time you actually cook the recipe
Connection A saved set of credentials Airflow uses to talk to an external system — your PostgreSQL database The phone number saved in your contacts — you dial the name, not the digits
Schedule A cron expression or preset that tells Airflow when to trigger a DAG automatically The alarm clock — set it once, it fires every time

What Airflow looks like in your stack

In this stack, Airflow runs inside two Docker containers — airflow-webserver (the UI at localhost:8080) and airflow-scheduler (the engine that reads DAG files and triggers runs). The scheduler watches the airflow/dags/ folder. Any Python file you put there becomes a DAG. No restart required — Airflow picks it up automatically within 30 seconds.

What is a cron expression? Cron is a time-based job scheduler that has been around since the 1970s. It uses a 5-field expression to define when something should run:

* * * * * — minute · hour · day-of-month · month · day-of-week

Examples:
0 */6 * * * — every 6 hours (at minute 0: 00:00, 06:00, 12:00, 18:00)
0 1 * * * — every day at 1:00am
0 2 * * * — every day at 2:00am
None — never runs automatically — manual trigger only

How Cosmos connects dbt to Airflow

Without Cosmos, running dbt from Airflow means writing a BashOperator task that executes dbt run as a shell command. That works — but it's a black box. Airflow sees one task: "run dbt." If 8 of your 12 staging models succeed and 4 fail, Airflow just reports the whole thing as failed. You have to open the logs and hunt for what went wrong.

Cosmos changes this. It reads your dbt project — the models, the tests, the dependencies between them — and converts each dbt model into its own individual Airflow task. Instead of one opaque task, you get a full task graph where every model is visible, every test is trackable, and a failure in one model doesn't hide which model caused it.

What is DbtTaskGroup? DbtTaskGroup is the core Cosmos pattern used in this project. It takes a dbt selector (like staging or marts) and generates a group of Airflow tasks — one per model — with all the correct dependencies already wired up from dbt's own lineage graph.

You pass it four things:
project_config — where the dbt project lives on disk
profile_config — how dbt connects to PostgreSQL (the Airflow connection)
execution_config — which dbt binary to use (the virtual environment path)
render_config — which models to include (the selector)

Cosmos handles the rest. It parses the dbt manifest, builds the task graph, and executes each model in the correct order.

Why one task per model matters

When every model is its own Airflow task, you get three things you cannot get from a single dbt run shell command:

  • Precise failure visibility — Airflow shows exactly which model failed, not just that "dbt failed." You can click into the task, read the logs, fix the model, and rerun just that task without rerunning the whole pipeline.
  • Partial reruns — if stg_revenue_account_daily fails at 1am, you can fix it and trigger just that task from the Airflow UI instead of rerunning all 12 staging models.
  • Task-level history — every model run is logged with its duration, status, and output. Over time this becomes a performance baseline — you can see if a model is getting slower as data volumes grow.
How does Cosmos know the model order? dbt generates a file called manifest.json every time you run dbt compile or dbt run. This file contains the full dependency graph for every model in the project — which models each one reads from, which tests apply to each model, and the execution order. Cosmos reads this manifest and uses it to wire up the Airflow task dependencies automatically. You never define the order manually.

The automated pipeline

This runbook sets up three DAGs. Each one targets a different layer of the medallion architecture and runs on its own schedule:

DAG schedule
silver_layer_staging Every 6 hours — 00:00, 06:00, 12:00, 18:00
silver_layer_staging_daily Every day at 1:00am
gold_layer_marts_daily Every day at 2:00am

The timing is intentional. Silver staging runs every 6 hours to keep the staging layer fresh throughout the day. The daily revenue staging model runs at 1am when new daily billing records are available. Gold marts run at 2am — after both silver pipelines have completed — so they always build on top of fully refreshed silver data.

DAGCosmos selectorModelsWhat it does
silver_layer_staging staging 12 staging views Rebuilds all staging views from bronze. Fast — views don't store data, they just refresh the SQL definition.
silver_layer_staging_daily intermediate 2 incremental tables Runs the incremental intermediate models — processes only new rows since the last run.
gold_layer_marts_daily marts 10 mart tables Rebuilds all gold mart tables. Runs after both silver pipelines so marts always reflect the latest data.
What about bronze and seeds? Bronze ingestion and seed loading are handled by separate DAGs already in the repo — bronze_layer_ingestion_manual.py and seed_layer_region_mapping.py. Both are manual-trigger only. Bronze data is loaded when new source data arrives, not on a fixed schedule. Seeds only need to run when the CSV changes. This runbook focuses on the three DAGs that automate the dbt transformation pipeline.
1

Verify Cosmos is installed

Cosmos was included in airflow/requirements.txt when you built the stack in Runbook 1. Confirm it is available inside the Airflow container:

docker exec airflow-scheduler pip show astronomer-cosmos
You see Name: astronomer-cosmos and Version: 1.13.1.

Also confirm the dbt virtual environment is in place — Cosmos needs to know exactly which dbt binary to call:

docker exec airflow-scheduler ls /usr/local/airflow/dbt_venv/bin/dbt
You see /usr/local/airflow/dbt_venv/bin/dbt — no "No such file" error.
Why does Cosmos need a separate dbt binary path? dbt is installed in its own virtual environment (dbt_venv) to keep its dependencies separate from Airflow's. If both shared the same Python environment, their package requirements would conflict — different versions of the same library required by each tool. Cosmos accepts an ExecutionConfig that points directly to the dbt binary inside the virtual environment, bypassing any conflict entirely.

Cosmos connects to PostgreSQL through an Airflow connection — a named set of credentials stored in the Airflow metadata database. The DAG files reference this connection by name (postgres_default) rather than hardcoding credentials.

Open Airflow at localhost:8080 and log in with admin / admin.

Navigate to Admin → Connections and click the + button to add a new connection. Fill in the fields exactly as shown:

The connection postgres_default appears in the connections list with type postgres.
Why is the host postgres and not localhost? Airflow and PostgreSQL both run inside Docker on the same network. Inside Docker, containers find each other by container name — not by localhost. The PostgreSQL container is named postgres in docker-compose.yml. From outside Docker (your Mac Terminal or pgAdmin), you use localhost:5432. From inside Docker (Airflow), you use postgres:5432.

Test the connection from the Terminal to confirm Airflow can reach PostgreSQL:

docker exec airflow-scheduler airflow connections test postgres_default
You see Connection tested successfully.
3

Add the DAG files

Why do the production schemas have a prod_ prefix? When dbt runs, it combines the target schema from profiles.yml with the custom schema from dbt_project.yml. Since the production target uses schema: prod in profiles.yml and the models are configured with +schema: staging_silver and +schema: mart_gold, dbt produces:

prod + staging_silver → prod_staging_silver
prod + mart_gold → prod_mart_gold

This is dbt's default behaviour. The DAG files below use these exact schema names so Airflow writes models to the correct production schemas. When you connect Metabase later, point it at prod_mart_gold.

The three DAG files live in airflow/dags/ inside the repo. Confirm they are there:

cd ~/Documents/btg-case-studies/resource-utilization
ls airflow/dags/
You see silver_layer_staging.py, silver_layer_staging_daily.py, and gold_layer_marts_daily.py in the list.

If any are missing, create them now. Open VS Code:

code airflow/dags/silver_layer_staging.py

Paste this exactly, then save with ⌘ + S:

from datetime import datetime
from pathlib import Path
from airflow import DAG
from airflow.operators.empty import EmptyOperator
from cosmos import DbtTaskGroup, ProjectConfig, ProfileConfig, ExecutionConfig, RenderConfig
from cosmos.profiles import PostgresUserPasswordProfileMapping

DBT_PROJECT_PATH = Path("/opt/airflow/dbt")
DBT_EXECUTABLE  = Path("/usr/local/airflow/dbt_venv/bin/dbt")

profile_config = ProfileConfig(
    profile_name="resource_utilization",
    target_name="prod",
    profile_mapping=PostgresUserPasswordProfileMapping(
        conn_id="postgres_default",
        profile_args={"schema": "prod_staging_silver"},
    ),
)
    start_date=datetime(2026, 1, 1),
    schedule_interval="0 */6 * * *",   # every 6 hours
    catchup=False,
    tags=["silver", "staging", "cosmos"],
) as dag:

    start = EmptyOperator(task_id="start")
    end   = EmptyOperator(task_id="end")

    staging = DbtTaskGroup(
        group_id="staging_models",
        project_config=ProjectConfig(DBT_PROJECT_PATH),
        profile_config=profile_config,
        execution_config=ExecutionConfig(dbt_executable_path=DBT_EXECUTABLE),
        render_config=RenderConfig(select=["staging"]),
    )

    start >> staging >> end

What does catchup=False mean? When a DAG has a schedule and a start_date in the past, Airflow by default tries to run it for every missed interval between the start date and now — this is called backfill. catchup=False disables this. When you first enable a scheduled DAG, it runs once at the next scheduled time rather than immediately running 50 times to catch up to today.

Verify DAGs appear in Airflow

Airflow's scheduler watches the dags/ folder and picks up new files automatically. Wait 30 seconds, then open localhost:8080.

You see all three DAGs listed: silver_layer_staging, silver_layer_staging_daily, and gold_layer_marts_daily. All three show a pause icon — they are paused by default.
Why are new DAGs paused by default? Airflow pauses every new DAG automatically so it does not start running before you have had a chance to review it. You must explicitly unpause a DAG to enable its schedule. This is a safety mechanism — it prevents a misconfigured DAG from firing immediately on every missed interval since its start date.

If a DAG does not appear after 60 seconds, check for import errors:

docker exec airflow-scheduler airflow dags list-import-errors
The output is empty — no import errors.

You can also check that Airflow parsed the DAG files correctly:

docker exec airflow-scheduler airflow dags list | grep -E "silver|gold"
You see all three DAG IDs listed with their schedules.

Trigger silver staging manually

Before enabling the schedule, trigger the pipeline manually to confirm everything runs end to end. Start with the silver staging DAG — it has no dependencies on the other two.

In the Airflow UI, click silver_layer_staging to open it. Click the ▶ Trigger DAG button (top right). Confirm when prompted.

Or trigger from the Terminal:

docker exec airflow-scheduler airflow dags trigger silver_layer_staging
The DAG run appears in the Runs column. Status moves from queued → running → success.

Click into the DAG run to see the task graph. You should see one task per staging model — all 12 — with their dependency arrows visible. Click any task to see its logs.

All 12 staging model tasks show green (success). The start and end tasks are also green.
If a task shows red (failed) — click the failed task → Log. The dbt error message will be in the log output. Common causes: the bronze table the model reads from is empty, or the Airflow connection credentials are wrong. Fix the issue and click Clear on the failed task to rerun just that task without rerunning the whole DAG.

Now trigger the intermediate DAG:

docker exec airflow-scheduler airflow dags trigger silver_layer_staging_daily
Both intermediate model tasks show green.

Then trigger the gold marts DAG:

docker exec airflow-scheduler airflow dags trigger gold_layer_marts_daily
All 10 mart model tasks show green.
The full pipeline ran successfully through Airflow. Bronze → silver staging → silver intermediate → gold marts — all orchestrated by Cosmos, with every model visible as its own task.

Enable the scheduled DAGs

All three manual runs succeeded. Enable the schedules so the pipeline runs automatically going forward.

In the Airflow UI, toggle the pause switch next to each DAG from off to on. Or do it from the Terminal:

docker exec airflow-scheduler airflow dags unpause silver_layer_staging
docker exec airflow-scheduler airflow dags unpause silver_layer_staging_daily
docker exec airflow-scheduler airflow dags unpause gold_layer_marts_daily
All three DAGs show a green active indicator in the Airflow UI. The Next Run column shows the next scheduled execution time for each.
What happens if the stack is not running at the scheduled time? Airflow will run the missed DAG on the next scheduled interval once the stack comes back up — but only if catchup=False is set, which it is in all three DAGs. This means one catch-up run at the next scheduled time, not a backlog of every missed run.

Confirm the schedule is active by checking the next run time for each DAG:

docker exec airflow-scheduler airflow dags next-execution silver_layer_staging
docker exec airflow-scheduler airflow dags next-execution silver_layer_staging_daily
docker exec airflow-scheduler airflow dags next-execution gold_layer_marts_daily
Each command returns a future timestamp matching the expected schedule.

Troubleshooting

DAG does not appear in Airflow after 60 seconds ▼

Check for import errors first:

docker exec airflow-scheduler airflow dags list-import-errors

The error message will point to the line in the DAG file that caused the problem — usually a missing import, a syntax error, or a wrong file path. Fix the DAG file, save it, and wait 30 seconds.

Task fails with "Could not connect to postgres_default" ▼

The Airflow connection was not saved correctly. Go to Admin → Connections, click postgres_default, and verify the host is postgres (not localhost) and the port is 5432. Test it from the Terminal:

docker exec airflow-scheduler airflow connections test postgres_default
Task fails with "dbt not found" or "No such file or directory" ▼

The dbt virtual environment path in the DAG file doesn't match where dbt is actually installed. Verify the correct path:

docker exec airflow-scheduler ls /usr/local/airflow/dbt_venv/bin/dbt

If the path is different from /usr/local/airflow/dbt_venv/bin/dbt, update the DBT_EXECUTABLE variable in all three DAG files to match.

Task fails with "manifest.json not found" ▼

Cosmos reads the dbt manifest to build the task graph. The manifest is generated when you run dbt compile or any dbt command. Run this to generate it:

cd ~/Documents/btg-case-studies/resource-utilization
dbt compile

Then retrigger the DAG.

DAG run shows success but gold tables are empty ▼

The mart DAG ran before the intermediate DAG finished — gold models found no data to build on. This can happen on the first manual run if you trigger all three at once. The fix is to trigger them in order: silver staging first, wait for success, then intermediate, wait for success, then gold. The scheduled runs handle this automatically through the timing offsets (1am vs 2am).

Airflow scheduler is not picking up DAG file changes ▼

The scheduler checks the DAGs folder every 30 seconds by default. If changes are not picked up after 60 seconds, restart the scheduler:

docker compose restart airflow-scheduler
How do I rerun a single failed task without rerunning the whole DAG? ▼

In the Airflow UI, click the DAG run → click the failed task (shown in red) → click Clear Task. Airflow will rerun only that task and any downstream tasks that depend on it. Tasks that already succeeded are not touched.

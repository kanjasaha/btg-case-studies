The full pipeline

This is what you are about to run — data flowing from bronze all the way through to gold:

LayerSchemaModelsWhat they do

Bronze raw_bronze 10 tables Already loaded — raw data from the Airflow DAG and data population scripts
Staging staging_silver 12 views Clean, rename, and standardize each bronze table — one model per source table
Intermediate staging_silver 2 incremental tables Heavy aggregations that mart models build on top of — token usage per minute, revenue per day
Marts mart_gold 10 tables Business-ready aggregations — token usage and revenue by model and customer, weekly/monthly/YTD
What is an incremental model? A regular dbt model rebuilds the entire table from scratch every run. An incremental model only processes new rows — rows that arrived since the last run. This is much faster for large tables.

Both intermediate models use delete+insert incremental strategy — on each run, dbt deletes the last 7 days of data and reinserts it fresh. This handles late-arriving data correctly without rebuilding everything.

Understanding dbt compile

What does dbt compile actually do? Your dbt SQL files contain Jinja templating — things like {{ ref('stg_orders') }} and {% if target.name != 'prod' %}. dbt compile translates all of that into plain SQL and saves it to target/compiled/ on your laptop. Nothing happens in the database.

This is useful when you want to:
— Check what SQL dbt will actually run before running it
— Debug why a Jinja expression is not working
— Copy the compiled SQL and paste it into pgAdmin to test manually
— See which schema a ref() resolves to in each target 
# Compile one model and look at the output
dbt compile --select stg_token_usage_open_source

# The compiled SQL appears here — open it in VS Code
code target/compiled/resource_utilization/models/staging/stg_token_usage_open_source.sql

Understanding --defer and the prod manifest

The problem dbt --defer solves. When you run a mart model, dbt needs all its upstream staging and intermediate models to exist first. Without --defer, you would have to rebuild every upstream model in your personal schema before running the one you actually want to test. That wastes time and compute.

--defer tells dbt: "for any model that doesn't exist in my personal schema, just read from the prod version instead."

So if you are only changing fct_orders, dbt reads upstream models from prod and only builds fct_orders in your personal schema. You only build what you changed.
Critical: --defer does not copy prod data into your schema. The output of your model always lands in your personal schema (dbt_yourname_mart_gold). What changes is what your model reads as input — it reads from prod tables instead of rebuilding them first.

Example — you change fct_orders which joins on stg_customers:
— Without --defer: you must first build stg_customers in your personal schema, then run fct_orders
— With --defer: dbt reads prod_staging_silver.stg_customers directly, runs your changed fct_orders logic against it, and writes the result to dbt_yourname_mart_gold.fct_orders

Your personal schema gets one new table. It does not become a copy of prod.
When to use --defer against prod (the full prod dataset). By default, --defer reads from your prod schemas — which have full data. This is useful when something looks wrong in CI or your sampled dev data, and you need to validate your model against real, full-scale data before merging. Be aware: running your model against 500M rows takes longer and costs more compute. Use it selectively — not for every run.
What is the prod manifest? Every time dbt runs, it generates a file called manifest.json in the target/ folder. This file is a complete snapshot of every model, test, and source in your project — including which schemas they live in.

For --defer to work, you need a copy of the prod manifest so dbt knows where to find prod tables. Save it to a folder called prod-state/: 
# Create the folder to hold the prod manifest
mkdir -p prod-state

# Copy the latest manifest as if it were from prod
# (In a real team this would be downloaded from S3 or a shared location)
cp target/manifest.json prod-state/manifest.json
The efficient personal development loop:
1. Make a change to a model
2. Run only what changed plus downstream:
dbt build --select state:modified+ --defer --state ./prod-state/
3. All tests run on your changed models using real prod data for unchanged upstream
4. Fix any failures, repeat
5. When all tests pass, open a Pull Request — CI will run the same command automatically

Running this locally before opening your PR means CI almost always passes on the first try.

Establish a clean baseline before doing any work

Why run a full build after every pull? Before you start any development work you need to confirm the project is healthy as it stands. If someone committed broken code to main overnight — a model that does not compile, a test that fails, a broken ref() — you need to know before you write a single line of SQL. Without this check you might spend an hour developing a new model only to discover at PR time that the project was already broken before you touched anything.

dbt build --target dev runs the full project — compile, run, and test — against the shared dev schema. If it passes the project is healthy and you have a clean baseline. If it fails something is broken in main and needs to be fixed before anyone starts new work.

First ensure your packages are in sync with packages.yml. If someone added or updated a package since your last pull, your local dbt_packages/ would be out of date:

cd ~/Documents/btg-case-studies/resource-utilization
dbt deps
You see Installed N package(s) with no errors.

Now run the full project build against the dev schema:

dbt build --target dev
What does dbt build do? dbt build is a single command that runs three things in order:

1. Compile — resolves all Jinja templates ({{ ref() }}, {{ source() }}, {% if is_incremental() %}) and builds the full dependency graph. Writes compiled SQL to dbt/target/compiled/. Nothing runs in the database yet.

2. Run — executes the compiled SQL against the database. Builds views in staging, incremental tables in intermediate, tables in marts. Also runs dbt seed for any seed files.

3. Test — runs all schema tests (not_null, unique, accepted_values) and all custom SQL tests against the models it just built. If any test fails, dbt reports which rows violated it.

This is also exactly what CI runs on every Pull Request and what Airflow runs in production — the only differences are the target (ci or prod) and the model selector (state:modified+ for CI, no selector for full prod builds).
If dbt build --target dev fails — the project has a problem in main that predates your work. Do not start developing. Check what failed, find the commit that broke it, and fix it first. Run dbt build --target dev again to confirm it passes before continuing.
dbt build --target dev completes with all models showing OK and all tests showing PASS. The dev schemas (dev_staging_silver, dev_mart_gold) are now populated with a fresh full build. You have a clean baseline — safe to start development work in your personal schema.
Refreshing data without recompiling — --no-compile

Once you have done the baseline build and confirmed nothing has changed in the codebase, subsequent data refreshes during the same day do not need to recompile. Use the --no-compile flag to skip compilation and execute the already-compiled SQL directly:

dbt run --no-compile --target dev

dbt skips Jinja resolution and the dependency graph rebuild and goes straight to executing the SQL in dbt/target/compiled/.

Only safe when:
— You have already done a full compile at least once (target/compiled/ exists)
— No code has changed since the last compile — if a teammate committed a model change, --no-compile would silently run the old SQL

Practical pattern:
Morning pull → dbt deps + dbt build --target dev (full baseline)
Data refresh later in the day → dbt run --no-compile --target dev (execute only)

Section A — Models

How the models folder is structured. Every SQL file in dbt/models/ is a model — a transformation that reads from sources or other models and writes an output. Three layers, each in its own subfolder:

models/staging/ — 12 models. Each reads from one bronze table, cleans and renames columns, writes a view.
models/intermediate/ — 2 models. Heavy aggregations between staging and marts. Incremental tables.
models/marts/revenue/ — 6 models. Business-ready revenue outputs.
models/marts/token_usage/ — 4 models. Business-ready token usage outputs.

Two YAML files govern the entire models folder:
models/staging/sources.yml — declares the bronze source tables dbt reads from.
models/staging/schema.yml, models/intermediate/schema.yml, models/marts/*/schema.yml — document and test each model's output columns.

What about the analyses/ folder?
dbt also has an analyses/ folder (created by dbt init) for SQL queries you want to version control and compile with dbt but never materialise into the database. Any .sql file in analyses/ gets Jinja compiled — {{ ref() }} and {{ source() }} resolve to real table paths — but dbt never executes it. You copy the compiled SQL from target/compiled/analyses/ and run it manually in pgAdmin.

This is useful for ad hoc exploratory queries, one-off audits, or diagnostic queries that need to reference production tables via ref() but should never become permanent models. This project does not use it — all SQL that produces output lives in models/.

config() parameters — full reference

What is required vs optional in config() — and what are the defaults?

No parameter is strictly required. The config() block itself is optional — if you omit it entirely dbt uses defaults from dbt_project.yml and its own built-in defaults. The config() block in a model file always overrides the project-level setting.

{{
    config(

        materialized='incremental',
        # OPTIONAL. Default: 'view'
        # Options: 'view', 'table', 'incremental', 'ephemeral'
        # Can be set for a whole folder in dbt_project.yml.
        # The config() block in the file overrides the project-level setting.

        unique_key=['account_id', 'revenue_date'],
        # CONDITIONALLY REQUIRED — only when materialized='incremental'
        # Without it dbt cannot identify which rows to replace.
        # No default — dbt will error if incremental and unique_key is missing.

        incremental_strategy='delete+insert',
        # OPTIONAL when materialized='incremental'.
        # Default: depends on the database adapter.
        #   dbt-postgres  → 'merge'
        #   dbt-snowflake → 'merge'
        #   dbt-bigquery  → 'merge'
        # Always set this explicitly — do not rely on adapter defaults.

        on_schema_change='append_new_columns',
        # OPTIONAL. Default: 'ignore'
        # 'ignore'             → new columns are silently dropped — data is lost
        # 'append_new_columns' → new columns are added to the existing table
        # 'sync_all_columns'   → adds new columns, removes deleted columns
        # 'fail'               → errors if schema changes at all
        # Always set this explicitly on incremental models.

        schema='mart_gold',
        # OPTIONAL. Default: target schema from profiles.yml
        # combined with any folder-level +schema from dbt_project.yml.
        # Only needed to override the project-level schema setting.

        alias='revenue_monthly',
        # OPTIONAL. Default: the SQL filename without .sql
        # e.g. token_usage_model_monthly.sql → token_usage_model_monthly
        # Only needed if you want a different name in the database.

        tags=['revenue', 'monthly', 'mart'],
        # OPTIONAL. Default: no tags.
        # Can also be set at folder level in dbt_project.yml.
        # Used for: dbt run --select tag:revenue

        post_hook='GRANT SELECT ON {{ this }} TO business_user',
        # OPTIONAL. Default: no hooks.
        # Runs after the model builds successfully.
        # {{ this }} resolves to the full table reference e.g.
        # prod_mart_gold.token_usage_model_monthly
        # pre_hook runs before the model builds.

        contract={'enforced': True},
        # OPTIONAL. Default: {'enforced': False}
        # When True, dbt compares output column names and types against
        # schema.yml and fails the run if they do not match exactly.
        # Requires all columns to be declared with data_type in schema.yml.

        access='public'
        # OPTIONAL. Default: 'protected'
        # 'protected' → only models in the same project can ref() this model
        # 'public'    → models in other dbt projects can ref() this model
        # 'private'   → only models in the same subfolder group can ref() this
    )
}}

The three you should always set explicitly on incremental models — even though they are optional, the defaults are silent and can cause data loss:

unique_key — required for incremental, dbt errors without it
incremental_strategy — do not rely on the adapter default, be explicit
on_schema_change — default is 'ignore' which silently drops new columns

Open dbt/models/staging/sources.yml. Read the inline comments:

# sources.yml tells dbt about the raw bronze tables it reads from.
# Without this file dbt cannot build the dependency graph — it would not
# know that stg_customer_details depends on raw_bronze.customer_details.
# sources.yml is for tables dbt does NOT own (loaded by Airflow).
# schema.yml is for tables dbt DOES own (models dbt creates itself).

version: 2   # required header — tells dbt this file uses the current format

sources:
  - name: raw_bronze          # the group name used in {{ source('raw_bronze', ...) }} calls
    schema: raw_bronze        # the PostgreSQL schema where these tables live
    description: "Bronze layer — raw ingested data from source systems"

    # Default freshness applied to ALL tables in this source group.
    # dbt source freshness checks the loaded_at column and compares to now().
    # warn_after: raises a warning but lets the run continue.
    # error_after: fails the run entirely.
    freshness:
      warn_after:  {count: 2, period: day}
      error_after: {count: 3, period: day}

    # The column dbt uses to measure freshness on every table.
    # Every bronze table has loaded_at — dbt checks this against now().
    loaded_at_field: loaded_at

    tables:

      # ── customer_details ────────────────────────────────────────────────────
      - name: customer_details
        description: "Customer and account information from CRM systems"
        # Override the default freshness — customer data changes rarely (monthly)
        # so a 30-day window is appropriate. Inference data changes hourly.
        freshness:
          warn_after:  {count: 30, period: day}
          error_after: {count: 60, period: day}
        columns:
          - name: account_id
            description: "Unique account identifier"
            tests:
              - not_null   # test runs on the raw bronze table before any transformation
              - unique     # catches duplicate account_ids in the source before they reach staging

      # ── inference_user_token_usage_open_source ───────────────────────────────
      - name: inference_user_token_usage_open_source
        description: "Token usage events for open source model inference"
        # Near-real-time table — warn if more than 2 hours old, error after 4.
        # Tighter window than customer data because inference data arrives continuously.
        freshness:
          warn_after:  {count: 2, period: hour}
          error_after: {count: 4, period: hour}
        columns:
          - name: request_id
            tests:
              - not_null
              - unique   # every inference request must have a unique ID

      # ── (remaining tables follow the same pattern) ────────────────────────
      # config tables (model_dimensions, region_availability) use 30/60-day
      # windows because they are updated manually via the Airflow config DAG.
      # resource tables (utilization, inventory, allocation) use 1–12 hour
      # windows because they are snapshots taken on a regular schedule.
      # quota and revenue tables use 1–2 day windows.

# and will run perfectly without it. However it is required by best practice
# for three reasons:
#
#   1. Tests — without schema.yml no not_null, unique, or accepted_values
#              tests run. Bad data reaches your marts silently.

#

config() block — full parameter reference

Every dbt model can have a config() block at the top that controls how dbt builds it. Here is a complete example using every available parameter, followed by all options for each one:

{{
    config(
        -- Controls how dbt materialises this model in the database
        materialized='incremental',

        -- The column(s) that uniquely identify one row.
        -- dbt uses this to find and replace existing rows on each run.
        -- Single column or composite key (list).
        unique_key=['account_id', 'revenue_date'],

        -- How dbt handles existing rows when new data arrives.
        incremental_strategy='delete+insert',

        -- What dbt does if you add or remove a column from the model.
        on_schema_change='append_new_columns',

        -- Override the schema from dbt_project.yml for this model only.
        schema='mart_gold',

        -- Override the table/view name (default = the SQL filename).
        alias='revenue_monthly',

        -- Labels for filtering: dbt run --select tag:revenue
        tags=['revenue', 'monthly', 'mart'],

        -- SQL to run after the model builds.
        -- {{ this }} resolves to the fully qualified table name.
        -- Used here to grant read access to the business_user role.
        post_hook='GRANT SELECT ON {{ this }} TO business_user',

        -- Enforce that output column names and types match schema.yml exactly.
        -- Fails the run if there is a mismatch — prevents silent breaking changes.
        contract={'enforced': True},

        -- Controls whether other dbt projects can ref() this model.
        access='public'
    )
}}
All available options for each parameter

materialized
'view' — default. Recreated as a SQL view on every run. No data stored — the query runs on demand. Used for staging models.
'table' — dropped and rebuilt from scratch on every run. All data reprocessed every time. Used when full rebuilds are acceptable.
'incremental' — builds a persistent table. On first run processes all data. On subsequent runs only processes new or changed rows. Used for large tables where full rebuilds are too slow.
'ephemeral' — never written to the database. Inlined as a CTE into any model that refs it. Useful for intermediate logic you want to reuse without creating a table.

unique_key (incremental only)
'account_id' — single column string
['account_id', 'revenue_date'] — composite key as a list. The combination of these columns identifies one unique row. dbt uses this to find existing rows to delete before reinserting.

incremental_strategy (incremental only)
'delete+insert' — deletes existing rows matching the unique_key, then reinserts. Handles late-arriving corrections correctly. Default for PostgreSQL.
'merge' — upsert: updates existing rows if they match, inserts if they do not. More efficient than delete+insert when updates are rare.
'append' — insert only, no deduplication. Fastest but can create duplicate rows if the same data arrives twice. Safe only when source data is guaranteed unique.

on_schema_change (incremental only)
'ignore' — default. New or removed columns are silently ignored. The existing table structure is unchanged.
'append_new_columns' — adds new columns to the existing table. Removed columns are left in place. Safe for additive changes.
'sync_all_columns' — adds new columns and removes deleted ones. Keeps the table in sync with the model exactly.
'fail' — errors immediately if the schema changes at all. Use when you want to force explicit migrations.

schema
Overrides the schema set in dbt_project.yml for this model only. Combined with the target prefix the model writes to dbt_yourname_mart_gold in personal and prod_mart_gold in prod.

alias
Overrides the table or view name. Without this, dbt uses the SQL filename as the table name. Use when you want a cleaner name in the database than the filename.

tags
'finance' — single tag as a string
['finance', 'monthly', 'mart'] — multiple tags as a list
Tags are used to select groups of models: dbt run --select tag:finance runs only models tagged finance. Also visible in dbt docs and the Airflow UI.

pre_hook
SQL to run before the model builds. Less common than post_hook.
pre_hook='TRUNCATE TABLE staging.temp_buffer'
pre_hook=['statement1', 'statement2'] — multiple statements as a list

post_hook
SQL to run after the model builds successfully. Most commonly used for table-level grants — schema-level grants are set in 001_setup_database.sql but table-level grants must run after the table exists.
post_hook='GRANT SELECT ON {{ this }} TO business_user'
{{ this }} resolves to the fully qualified table name at compile time e.g. prod_mart_gold.revenue_monthly.
post_hook=['GRANT SELECT ON {{ this }} TO business_user', 'GRANT SELECT ON {{ this }} TO analytics_engineer']

contract
{'enforced': True} — dbt checks that every column name and data type in the model output exactly matches what is declared in schema.yml. Fails the run if there is any mismatch. Used on public mart models to prevent silent breaking changes from reaching downstream consumers.
{'enforced': False} — default. No contract checking.

access

-- token_usage_model_monthly.sql
-- Aggregates minute-level token usage to monthly grain per model.
-- This is what Metabase queries for the monthly token usage dashboard.
-- Reads from int_token_usage_minute — never directly from staging.

{{

        materialized='incremental',
        -- No account_id here — this is a model-level mart not a customer-level mart.
        -- The customer-level equivalent is token_usage_customer_monthly.sql.
        unique_key=['model_variant', 'model_type', 'source_region',

        incremental_strategy='delete+insert',
        on_schema_change='append_new_columns'
    )

with monthly as (

    select
        model_variant, model_type, model_family, model_publisher,
        source_region, inference_region, inference_scope,
        territory, govcloud,

        -- Truncate each minute_timestamp to the first day of its month.
        -- ::date strips the time component → clean date output.
        date_trunc('month', minute_timestamp)::date          as month_start_date,

        -- Calculate the last day of the month by going forward 1 month then back 1 day.
        (date_trunc('month', minute_timestamp)
            + interval '1 month' - interval '1 day')::date  as month_end_date,

        request_count, total_tokens, input_tokens, output_tokens, error_count

    from {{ ref('int_token_usage_minute') }}

    {% if is_incremental() %}
        -- Look back one extra month to recalculate the current partial month
        -- if new data arrives after the last run.
        -- Without the - interval '1 month', today's data would be missed
        -- until the next full month boundary.
        where event_date >= (select max(month_start_date) from {{ this }})
                          - interval '1 month'
    {% endif %}

),

aggregated as (
    select
        model_variant, model_type, model_family, model_publisher,
        source_region, inference_region, inference_scope,
        territory, govcloud, month_start_date, month_end_date,

        -- mean_rpm: average requests per minute across all minutes in the month.
        -- int_token_usage_minute is already at minute grain — avg(request_count)
        -- gives the mean RPM without needing to know total minutes.
        round(avg(request_count), 2)                        as mean_rpm,
        max(request_count)                                  as peak_rpm,

        -- peak_to_mean ratio: high ratio = bursty traffic, low = steady.
        -- Useful for capacity planning. nullif prevents divide-by-zero.
        round(max(request_count)::numeric
            / nullif(avg(request_count), 0), 2)             as peak_to_mean_rpm,

        round(avg(total_tokens), 2)                         as mean_tpm,
        max(total_tokens)                                   as peak_tpm,
        round(max(total_tokens)::numeric
            / nullif(avg(total_tokens), 0), 2)              as peak_to_mean_tpm,

        sum(total_tokens)                                   as total_tokens,
        sum(input_tokens)                                   as total_input_tokens,
        sum(output_tokens)                                  as total_output_tokens,
        sum(request_count)                                  as total_requests,
        sum(error_count)                                    as total_errors,
        round(sum(error_count)::numeric
            / nullif(sum(request_count), 0) * 100, 2)       as error_rate_pct

    from monthly
    group by
        model_variant, model_type, model_family, model_publisher,
        source_region, inference_region, inference_scope,
        territory, govcloud, month_start_date, month_end_date
),

with_mom as (
    select
        a.*,

        -- LAG() window function: looks back one row (previous month)
        -- partitioned by model + region so we compare like-for-like.
        -- order by month_start_date ensures the previous row is actually
        -- the previous month, not any arbitrary earlier row.
        a.mean_rpm - lag(a.mean_rpm) over (
            partition by a.model_variant, a.model_type,
                         a.source_region, a.inference_region
            order by a.month_start_date
        )                                                   as mom_mean_rpm,
        -- mom = month over month. Same LAG pattern repeated for each metric.

        a.mean_tpm - lag(a.mean_tpm) over (
            partition by a.model_variant, a.model_type,
                         a.source_region, a.inference_region
            order by a.month_start_date
        )                                                   as mom_mean_tpm,

        a.total_tokens - lag(a.total_tokens) over (
            partition by a.model_variant, a.model_type,
                         a.source_region, a.inference_region
            order by a.month_start_date
        )                                                   as mom_total_tokens,

        a.total_requests - lag(a.total_requests) over (
            partition by a.model_variant, a.model_type,
                         a.source_region, a.inference_region

Section D — Macros and Tests

What is the macros folder? dbt/macros/ contains reusable Jinja functions that generate SQL. Instead of writing the same complex SQL pattern in multiple models, you write it once as a macro and call it by name. Macros run during compilation — before any SQL runs in the database — so they produce SQL text, not query results.

This project has 6 macros. Two work as a pair: add_prior_period_metrics calls safe_prior_period internally.

safe_prior_period.sql — annotated

Open dbt/macros/safe_prior_period.sql — this is the lower-level macro, read it first:

-- safe_prior_period.sql
-- Generates a LAG window function with a data-gap guard.
-- Called by add_prior_period_metrics — never called directly in models.

{% macro safe_prior_period(metric, partition_cols, date_col, period_type) %}
-- Arguments:
--   metric        : the column to look back on e.g. 'net_revenue'
--   partition_cols: list of columns to partition by e.g. ['model_variant', 'source_region']
--   date_col      : the date column to order by e.g. 'month_start_date'
--   period_type   : 'month', 'week', 'day' — controls the interval gap check

    case
        -- The guard: check whether the previous row is actually one period earlier.
        -- lag(date_col) looks back one row. If that row's date equals
        -- the current date minus one period, the data is contiguous — safe to use.
        -- If the data has a gap (e.g. Jan → Mar skipping Feb), the previous row
        -- would be Jan which does NOT equal Mar - 1 month = Feb.
        -- In that case we return 0 instead of a misleading prior-period value.
        when lag({{ date_col }}) over (
            partition by {{ partition_cols | join(', ') }}
            order by {{ date_col }}
        ) = {{ date_col }} - interval '1 {{ period_type }}'
        -- If the gap check passes — return the previous row's metric value
        then lag({{ metric }}) over (
            partition by {{ partition_cols | join(', ') }}
            order by {{ date_col }}
        )
        -- If the gap check fails (missing period) — return 0 not null.
        -- Returning null would make the change calculation null too,
        -- which is harder to handle in BI tools than a clean 0.
        else 0
    end
{% endmacro %}

        partition_cols,
        date_col,
        period_type

    -- period_type[0:3] takes the first 3 chars: 'month' → 'mon', 'week' → 'wee', 'day' → 'day'
    -- "," if not loop.last else "" adds a comma after every column except the last —
    -- avoids trailing comma SQL syntax errors when the macro is used mid-SELECT.

    {% endfor %}
{% endmacro %}

Understanding a staging model — stg_token_usage_open_source.sql

Open dbt/models/staging/stg_token_usage_open_source.sql to see a representative example. Every staging model follows the same three-CTE pattern:

1. config block
{{ config(materialized='view') }}
Tells dbt to create this as a PostgreSQL VIEW rather than a table. Views do not store data — they are saved SQL queries that run on demand. Staging models use views because they are cheap to rebuild and are read by intermediate models which store the actual data.

2. source CTE
select * from {{ source('raw_bronze', 'inference_user_token_usage_open_source') }}
The {{ source() }} function is how dbt references a bronze table. It tells dbt this model depends on the raw_bronze source, which is defined in sources.yml. dbt uses this to build the dependency graph and run source freshness checks.

3. filtered CTE
Removes rows with null values in critical columns — request_id, account_id, model_variant and others. These nulls would cause silent errors in downstream aggregations. Filtering here means every model downstream can trust these columns are never null.

4. renamed CTE
The main transformation block. Key patterns:
— Column renames: input_token as input_tokens — standardises plural naming across the project
— Hardcoded value: 'open_source' as model_type — tags every row with its model type so the table can be unioned with the proprietary equivalent
— coalesce(cache_read_token, 0) — replaces nulls with zero so arithmetic never produces null results
— input_token + output_token + coalesce(...) as total_tokens — derives a calculated column from raw columns
— date_trunc('minute', event_timestamp) as minute_timestamp — adds time bucketing columns that intermediate models aggregate on
— date(event_timestamp) as event_date — extracts just the date for daily aggregations

5. Final select
select * from renamed
Always selects from the last CTE. The CTE chain makes the logic readable — each step has a clear name and purpose.

Run intermediate models

What are intermediate models?Intermediate models sit between staging and marts. They do the heavy aggregation work that would be too expensive to repeat in every mart model. They are incremental — on each run, they only process new data rather than rebuilding from scratch.

Both intermediate models live in staging_silver — they are not business-facing outputs, just efficient stepping stones for the mart models above them.
ModelWhat it does
int_token_usage_minute Combines open source and proprietary token usage into one table, aggregated per minute per account per model per region. Mart models read from here instead of joining two large staging views directly.
int_revenue_daily Joins stg_revenue_account_daily with customer details to produce a clean daily revenue record enriched with customer segment and vertical. Calculates net revenue = gross - savings_plan - discount.
What does incremental mean in practice?The first time you run these models, dbt builds the full table from all available data. Every subsequent run, dbt checks the max date already in the table and only processes rows newer than that (minus a 7-day overlap for late-arriving data). Much faster than rebuilding everything each time.
Understanding an incremental model — int_revenue_daily.sql

Open dbt/models/intermediate/int_revenue_daily.sql. This model shows two important patterns: incremental materialisation and multi-source joining.

1. config block
materialized='incremental' — builds a persistent table, not a view. Only processes new rows on each run.
unique_key=['account_id', 'model_variant', 'source_region', 'revenue_date'] — the combination of these four columns uniquely identifies a row. dbt uses this to detect which rows already exist.
incremental_strategy='delete+insert' — on each run, dbt deletes existing rows that match the incoming data and reinserts them fresh. This handles late-arriving corrections correctly.
on_schema_change='append_new_columns' — if you add a new column to the model, dbt adds it to the existing table rather than failing.

2. is_incremental() block
{% if is_incremental() %}
where revenue_date >= (select max(revenue_date) from {{ this }}) - interval '7 days'
This block only runs on the second run onwards — not the first full build. {{ this }} refers to the table itself. It reads the most recent date already in the table and only pulls the last 7 days of data — a 7-day window to catch any late-arriving corrections. On the very first run, this block is skipped and the full history is built.

3. Multiple CTEs as building blocks
revenue — reads from the staging revenue model
customer — reads customer dimensions from staging
model_config — reads model family and publisher from config staging
region_mapping — reads territory and govcloud flag from the seed table
Each CTE selects only the columns it needs — not select * — which keeps joins clean and explicit.

4. enriched CTE — the join layer
Joins all four CTEs together using LEFT JOIN — meaning revenue rows are kept even if a customer or model has no matching config record. This is safer than an INNER JOIN which would silently drop revenue rows with missing dimensions.
net_revenue is derived here: total_gross_revenue - coalesce(savings_plan, 0) - coalesce(discount, 0). Derived columns are calculated at the intermediate layer so every mart model reads the same consistent value rather than each calculating it differently.

Understanding a macro — add_prior_period_metrics.sql

Open dbt/macros/add_prior_period_metrics.sql. A macro is a reusable Jinja function that generates SQL. Instead of writing the same LAG window function pattern in every mart model, you write it once as a macro and call it everywhere.

Macro signature
{% macro add_prior_period_metrics(metrics, partition_cols, date_col, period_type='day') %}
Takes three required arguments and one optional one: a list of metrics to compare, the columns to partition by, the date column to compare across, and the period type (defaults to 'day'). This single macro generates both the prior period value and the period-over-period change for any list of metrics.

{% for metric in metrics %}
Loops through the list of metrics passed in. For each metric it generates two columns — the prior period value and the change. This means calling the macro with 3 metrics generates 6 SQL columns automatically.

{{ safe_prior_period(...) }}
Calls another macro — macros can call other macros. safe_prior_period generates a LAG window function with null-safe handling. Nesting macros keeps each one focused on one job.

{{ metric.split('.')[-1] }}
Jinja string manipulation — extracts just the column name from a fully qualified reference like r.net_revenue, giving net_revenue for use in the output column name.

{{ "," if not loop.last else "" }}
Adds a comma after every generated column except the last — a common Jinja pattern to avoid trailing comma SQL errors.
Understanding a custom test — test_revenue_always_positive.sql

Run dbt snapshot

What are snapshots?Snapshots track how records change over time. Bronze tables are append-only — they never update existing rows. But some source data does change — a customer's segment, a model going inactive. A snapshot checks for changes on each run and adds a new row with timestamps showing when the change happened, giving you a full history of every state a record has ever been in.

This project has two snapshots, both writing to a snapshots schema:
SnapshotTracksStrategy
customer_details_snapshot Changes to customer segment, vertical, account size, and active status over time timestamp — checks the date_updated column. If it changed, a new snapshot row is added.
model_availability_snapshot When AI models go active or inactive in each region check — checks the is_region_active column. If it changed, a new snapshot row is added.
timestamp vs check strategy — what is the difference? timestamp — uses a column like updated_at to detect changes. Efficient because it only looks at rows where the timestamp changed.

check — compares specific columns directly. Use when the source doesn't have an updated_at column but you still want to track changes to specific fields.
Understanding a snapshot file — customer_details_snapshot.sql

Open dbt/snapshots/customer_details_snapshot.sql. Snapshot files use a special {% snapshot %} block instead of a regular SQL file.

{% snapshot customer_details_snapshot %} ... {% endsnapshot %}
The wrapper block that identifies this as a snapshot to dbt. The name inside the tags becomes the table name in the snapshots schema.

target_schema='snapshots'
Writes the snapshot table into the snapshots schema. Combined with the target prefix this becomes dbt_yourname_snapshots in your personal environment.

unique_key='account_id'
The column dbt uses to identify each record. When dbt sees the same account_id appear again with different values, it knows a change has occurred.

strategy='timestamp'
Uses the date_updated column to detect changes. dbt compares the date_updated value in the source against what it last saw. If it changed, the record has been updated.

updated_at='date_updated'
Points to the specific column that carries the last-updated timestamp in the source data.

invalidate_hard_deletes=True
If a customer is deleted from the source, dbt closes their snapshot row by setting dbt_valid_to to the current timestamp. Without this, deleted records would remain open in the snapshot forever as if still active.

The select statement
select ... from {{ ref('stg_customer_details') }}
Snapshots read from staging models — not directly from bronze. This means the snapshot captures already-cleaned data with standardised column names. Only the columns you select are tracked — customer_segment, customer_vertical, customer_account_size, is_active_customer — any change to these columns creates a new snapshot row.

What dbt adds automatically
dbt adds four columns to every snapshot table:
dbt_scd_id — unique ID for each snapshot row
dbt_updated_at — when this snapshot row was created
dbt_valid_from — when this version of the record became active
dbt_valid_to — when this version ended (null = currently active)

To find a customer's current state: WHERE dbt_valid_to IS NULL.
To see what a customer looked like on a specific date: WHERE dbt_valid_from <= '2025-06-01' AND (dbt_valid_to > '2025-06-01' OR dbt_valid_to IS NULL).

Run snapshots for the first time — this creates the initial snapshot of all current records:

dbt snapshot
Both snapshots show OK snapshotted. In pgAdmin, a snapshots schema appears with customer_details_snapshot and model_availability_snapshot tables inside.
How do snapshots work on subsequent runs?The second time you run dbt snapshot, dbt compares the current source data against the last snapshot. For any records that changed, it closes the previous row (sets dbt_valid_to to now) and inserts a new row with the current values. Records that didn't change are untouched. This is called Slowly Changing Dimension Type 2 (SCD2) — a standard data warehousing pattern.

Troubleshooting

dbt command not found ▼

The shell function hasn't been loaded yet. Run:

source ~/.zshrc

If it still doesn't work, check that the function is in ~/.zshrc: cat ~/.zshrc

TypeError: __init__() got an unexpected keyword argument 'buffer_size' ▼

You are running the dbt version installed directly on your Mac instead of the one inside Docker. The Mac version is old and incompatible. The fix is to make sure the shell function is loaded and taking priority:

source ~/.zshrc
type dbt

You should see dbt is a shell function. If you see dbt is /usr/local/bin/dbt or dbt is /Users/yourname/Library/Python/..., the Mac version is still winning. Check that the function definition is at the bottom of ~/.zshrc — it must come after any other dbt-related lines.

"relation does not exist" when running a model ▼

The bronze table the model reads from is empty or missing. Check:

docker exec -it postgres psql -U mds_user -d btg_resource_utilization -c "SELECT tablename FROM pg_tables WHERE schemaname = 'raw_bronze' ORDER BY tablename;"

If tables are missing, go back to Runbook 2 and re-run the bronze setup scripts.

dbt test fails ▼

dbt shows you which test failed and how many rows violated it. Open the test file in dbt/tests/ and run it manually in pgAdmin's Query Tool to see exactly which rows are causing the failure. This will point you to the data quality issue.

Incremental model shows 0 rows inserted ▼

This means there is no new data since the last run — which is expected if you run dbt twice on the same dataset. If you expect new data, check that new rows were actually inserted into the bronze table.

dbt snapshot fails with "column does not exist" ▼

The snapshot reads from a staging view that references a column that doesn't exist in the bronze table. Check that all data population scripts ran successfully and the bronze tables have the expected columns.

"macro not found" error ▼

The dbt packages haven't been installed. Run:

dbt deps

Then retry the failing command.

dbt run succeeds but tables not visible in pgAdmin ▼

Refresh pgAdmin — right-click the schema → Refresh. pgAdmin does not auto-refresh when new tables are created.

The medallion architecture

Before writing a single line of SQL, understand why we structure data the way we do. This pattern is used at companies like Databricks, Netflix, and Airbnb.

What is the medallion architecture?It organizes data into layers — each one more refined than the last. Think of refining gold: raw ore → remove impurities → pure gold ready to use.
SchemaLayerWhat lives hereWho writes tablesTable grants managed by
raw_bronzeBronzeRaw data exactly as it arrived — never modifiedIngestion scripts, Airflow DAGsSetup SQL script (static tables)
seedsSeedsStatic CSV lookup tables — region codes, product namesdbt seed onlydbt post-hook
staging_silverSilverCleaned, standardized datadbt models onlydbt post-hook — analytics_engineer, data_scientist, and selectively partner_dw_engineer
staging_silver_dsSilver DSData science model outputsData scientists onlyData scientist team
mart_goldGoldBusiness-ready aggregations and metricsdbt models onlydbt post-hook
The most important rule — bronze is append-only.Bronze data is never updated or deleted. New data is always added as new rows. You always have a full history, and can always rebuild downstream from bronze.
Why are silver and gold grants managed by dbt post-hooks?Table-level grants can only be given to tables that exist. When the setup script runs, silver and gold tables don't exist yet — dbt creates them when models run. A post-hook runs a GRANT automatically after each model builds, keeping permissions always in sync.

What does each dbt entry do? dbt/profiles.yml — contains your database password. Created in Step 3. Must never go to GitHub — share profiles.yml.example with teammates instead.

Create dbt_project.yml

What is dbt_project.yml?The main configuration file. Tells dbt the project name, where to find models, how to configure each layer, and which post-hooks to run after each model builds. Every dbt project must have exactly one of these in the dbt/ root folder.
YAML is indentation-sensitive.Use 2 spaces per level — never tabs. A single wrong space causes a syntax error. VS Code highlights YAML errors with a red underline.
cd ~/Documents/btg-case-studies/resource-utilization
code dbt/dbt_project.yml

Type this exactly, then save with ⌘ + S:

# dbt_project.yml
# Main configuration file. Every dbt project must have exactly one.

# Project name — must match profile: in profiles.yml
name: 'resource_utilization'
version: '1.0.0'

# Format version of this config file — always 2 for modern dbt
config-version: 2

# Which connection profile to use — defined in profiles.yml
profile: 'resource_utilization'

# Where dbt looks for each type of file
model-paths: ["models"]   # SQL transformation models
test-paths:  ["tests"]    # custom data quality tests
seed-paths:  ["seeds"]    # CSV lookup tables
macro-paths:     ["macros"]     # reusable SQL functions
snapshot-paths:  ["snapshots"]  # slowly changing dimension snapshots

# Where dbt writes compiled SQL — gitignored, never commit
target-path: "target"
clean-targets:
  - "target"
  - "dbt_packages"

# Seed configuration — controls how CSV files in seeds/ are loaded
seeds:
  resource_utilization:
    +schema: seeds            # write to the seeds schema in PostgreSQL
    +post-hook:               # grant access after loading
      - "GRANT SELECT ON {{ this }} TO analytics_engineer"
      - "GRANT SELECT ON {{ this }} TO data_scientist"

# Model configuration — controls how each folder of models behaves
models:
  resource_utilization:   # must match project name above

    staging:
      +schema: staging_silver   # write to staging_silver schema
      +materialized: view       # VIEW — virtual, always fresh, saves storage
      +post-hook:               # grant access after each model builds
        - "GRANT SELECT ON {{ this }} TO analytics_engineer"
        - "GRANT SELECT ON {{ this }} TO data_scientist"

    intermediate:
      +schema: staging_silver   # lives in staging_silver alongside staging models
      +materialized: incremental # INCREMENTAL — only processes new rows each run
      +post-hook:
        - "GRANT SELECT ON {{ this }} TO analytics_engineer"
        - "GRANT SELECT ON {{ this }} TO data_scientist"

    marts:
      +schema: mart_gold        # write to mart_gold schema
      +materialized: table      # TABLE — physically stored, faster for Metabase
      +post-hook:               # grant access after each model builds
        - "GRANT SELECT ON {{ this }} TO analytics_engineer"
        - "GRANT SELECT ON {{ this }} TO business_user"
name and profile — they must match each other. name: 'resource_utilization' is the project name.
profile: 'resource_utilization' tells dbt which entry in profiles.yml to use for the database connection.

These two values must be identical. Think of it as dbt looking up the project name in profiles.yml like a phonebook — if the names don't match, dbt cannot find its connection credentials and will error immediately.
model-paths, test-paths, seed-paths etc. — where dbt looks for files. Each path tells dbt where to find a specific type of file inside the dbt/ folder:

model-paths: ["models"] → look for .sql transformation files in dbt/models/
test-paths: ["tests"] → look for custom test SQL files in dbt/tests/
seed-paths: ["seeds"] → look for .csv lookup files in dbt/seeds/
macro-paths: ["macros"] → look for reusable SQL functions in dbt/macros/
snapshot-paths: ["snapshots"] → look for snapshot definitions in dbt/snapshots/

These match exactly the folders you created in Step 1. If you put a model file anywhere else, dbt will not find it.
target-path and clean-targets — where dbt writes generated files. target-path: "target" — dbt writes all compiled SQL and run artifacts into dbt/target/. This is why dbt/target/ is in your .gitignore — it is machine-generated output, not code you wrote.

clean-targets — the list of folders that dbt clean wipes when you want a completely fresh start. Running dbt clean deletes both target/ and dbt_packages/ so you can rebuild from scratch.
The + prefix — what it means. Throughout this file you see +schema, +materialized, +post-hook. The + prefix means the setting is a dbt configuration that applies to all models in that folder — not a subfolder name. Without +, dbt would think you are defining a new subfolder called schema or materialized rather than a configuration setting.
seeds block — how CSV lookup tables are loaded. +schema: seeds — when you run dbt seed, CSV files land in a schema called seeds. Combined with the generate_schema_name macro from Step 2b, this becomes dbt_alice_seeds in your personal environment and just seeds in production.

+post-hook — SQL that runs automatically after each seed loads. Here it grants read access to analytics_engineer and data_scientist roles. Because table-level grants can only be applied after the table exists, post-hooks are the right place for this — not the schema setup script.
models block — the most important section. For each folder of models it sets three things:

+schema — which custom schema to write to. Combined with generate_schema_name:
  staging_silver → personal: dbt_alice_staging_silver, prod: staging_silver
  mart_gold → personal: dbt_alice_mart_gold, prod: mart_gold

+materialized — how dbt physically creates the model in PostgreSQL:
ValueWhat it createsWhy this layer uses it
viewA virtual view — no data stored, always reads fresh from sourceStaging — lightweight, always current, cheap to rebuild
incrementalA table that only adds new rows each run — skips rows already processedIntermediate — large aggregations, processing full data every run would be slow
tableA full physical table — all rows stored, rebuilt completely each runMarts — fast for Metabase to query, worth the rebuild cost
+post-hook — SQL that grants read access after each model builds. Each layer grants to different roles:
  staging and intermediate → analytics_engineer, data_scientist
  marts → analytics_engineer, business_user

Business users only need mart access — they should never query raw staging views directly.
What does {{ this }} mean?Jinja templating — dbt's way of writing dynamic SQL. {{ this }} refers to the exact table or view just created by the current model. So after stg_customer_details builds, dbt runs GRANT SELECT ON staging_silver.stg_customer_details TO analytics_engineer automatically — keeping permissions always in sync without you having to write the table name manually.
How do the +schema values combine with profiles.yml to produce final schema names? dbt takes your schema from profiles.yml and appends the +schema from dbt_project.yml with an underscore. But in production you want clean schema names with no prefix. Step 2b adds a macro that controls this behaviour per environment.

Without the macro (dbt default):
  personal (schema: dbt_alice) + staging_silver → dbt_alice_staging_silver
  prod (schema: prod) + staging_silver → prod_staging_silver ← ugly

With the macro (Step 2b):
  personal → dbt_alice_staging_silver ← isolated ✅
  dev → dev_staging_silver ← isolated ✅
  ci → ci_staging_silver ← isolated ✅
  prod → staging_silver ← clean, no prefix ✅
What are snapshots? Snapshots track how a record changes over time. Bronze tables are append-only — they never update existing rows. But some source data does change, like a customer's segment or a model going inactive. A dbt snapshot checks for changes and records a new row with a timestamp whenever something changes, giving you a full history.

This project has two snapshots:
customer_details_snapshot — tracks when a customer's segment, size, or status changes over time.
model_availability_snapshot — tracks when models go active or inactive in each region.

Snapshots are run with dbt snapshot — covered in Runbook 3.

Create the schema name macro

What is a macro?A reusable piece of Jinja code that dbt calls automatically during every run. The generate_schema_name macro is a special built-in hook — dbt calls it every time it needs to figure out which schema to write a model into. By overriding it, you control exactly how schema names are generated for each environment.
Why do we need this? Without this macro, dbt always concatenates your target schema with the custom schema:
prod + staging_silver → prod_staging_silver

That is fine for dev isolation but ugly in production. This macro tells dbt:
— In personal, dev, ci: concatenate (so environments never collide)
— In prod: use only the custom schema (clean names for Metabase and business users) 
cd ~/Documents/btg-case-studies/resource-utilization
code dbt/macros/generate_schema_name.sql

Type this exactly, then save with ⌘ + S:

-- macros/generate_schema_name.sql
-- Controls how dbt names schemas in each environment.
--
-- How it works:
--   personal → dbt_alice_staging_silver  (your private isolated schemas)
--   dev      → dev_staging_silver        (shared team schemas, isolated)
--   ci       → ci_staging_silver         (automated PR test schemas, isolated)
--   prod     → staging_silver            (clean names, no prefix)
--
-- dbt calls this macro automatically on every run.
-- You never call it directly.

{% macro generate_schema_name(custom_schema_name, node) -%}

  {%- set default_schema = target.schema -%}
  {# default_schema comes from profiles.yml:
     personal → "dbt_alice"
     dev      → "dev"
     ci       → "ci"
     prod     → "prod"                    #}

  {%- if custom_schema_name is none -%}
    {# No +schema set in dbt_project.yml for this model.
       Just use the target schema as-is.
       personal → "dbt_alice"
       prod     → "prod"                  #}
    {{ default_schema }}

  {%- elif target.name == 'prod' -%}
    {# Production: use ONLY the custom schema — drop the prefix entirely.
       prod + staging_silver → "staging_silver"
       prod + mart_gold      → "mart_gold"      #}
    {{ custom_schema_name | trim }}

  {%- elif target.name in ['personal', 'dev', 'ci'] -%}
    {# Non-prod: prefix with target schema to isolate each environment.
       personal + staging_silver → "dbt_alice_staging_silver"
       dev      + staging_silver → "dev_staging_silver"
       ci       + staging_silver → "ci_staging_silver"
       personal + mart_gold      → "dbt_alice_mart_gold"
       dev      + mart_gold      → "dev_mart_gold"
       ci       + mart_gold      → "ci_mart_gold"            #}
    {{ default_schema }}_{{ custom_schema_name | trim }}

  {%- else -%}
    {# Safety fallback for any unknown target name.
       Concatenates like non-prod to avoid accidentally writing to wrong place. #}
    {{ default_schema }}_{{ custom_schema_name | trim }}

  {%- endif -%}

{%- endmacro %}

Create profiles.yml

What is profiles.yml?Where dbt stores database connection details. Separate from dbt_project.yml by design — project settings belong in the repo, connection details stay on your machine. Each target is a named connection pointing to a different environment.
profiles.yml contains your password — NEVER commit it to GitHub.It is blocked by .gitignore from Runbook 1. After creating it, always run git status to confirm it is invisible to Git.
profiles.yml lives inside dbt/ — not in ~/.dbt/Our Docker setup maps dbt/ to /opt/airflow/dbt inside the container, so dbt finds the file there automatically.
What are targets? Each target is a named connection profile pointing to a different environment. You will create four:

personal — your own private schemas, named after you. This is where you develop.
dev — shared team schemas for integration checking.
ci — used by GitHub Actions when running automated tests on Pull Requests.
prod — live schemas that Airflow writes to and Metabase reads from.

Because all four environments run in the same PostgreSQL instance on your Mac, the host and credentials are the same — only the schema names differ.

For your personal target (schema: dbt_alice):

For prod target (schema: prod):

This keeps every developer's schemas separate without any conflicts.

Create schemas and roles

What is a schema?A named container for tables inside a database. Think of the database as a filing cabinet and schemas as the drawers inside it.
What is a role?A job title in PostgreSQL. Controls what a user can do. We create them now — as the project grows, you grant roles to new users instead of managing permissions one by one.
Why are we creating schemas for four environments? As explained in Runbook 1, this project uses four environments: personal, dev, CI, and production. All four live in the same PostgreSQL database as separate schemas. Creating them all now means the database is ready for the full team workflow you will learn in Runbook 5.

Replace yourname with your actual first name (lowercase) everywhere below — for example dbt_alice if your name is Alice.

Step 3b — Create the Airflow DAG

What is a DAG?A DAG (Directed Acyclic Graph) is a Python file that defines a workflow — what tasks to run and in what order. Our DAG has two tasks: load model_configuration.json, then load model_region_availability.json. It only runs when you manually trigger it.
code airflow/dags/load_config_bronze.py

Type this exactly, then save with ⌘ + S:

"""
load_config_bronze.py
One-time DAG to load static JSON config files into bronze tables.
Trigger manually from the Airflow UI — never runs on a schedule.

Files:
  database/config/model_configuration.json
    → raw_bronze.config_model_dimensions (20 records)
  database/config/model_region_availability.json
    → raw_bronze.config_model_region_availability (41 records)
"""

import json
import os
from datetime import datetime

import psycopg2
from airflow import DAG
from airflow.operators.python import PythonOperator

DB_CONN = {
    "host":     "postgres",
    "port":     5432,
    "dbname":   "btg_resource_utilization",
    "user":     "mds_user",
    "password": "mds_password",
}

# Mapped from docker-compose volume: ./database → /opt/airflow/database
CONFIG_DIR = "/opt/airflow/database/config"


def load_model_configuration():
    """Load model_configuration.json → raw_bronze.config_model_dimensions"""

    filepath = os.path.join(CONFIG_DIR, "model_configuration.json")
    with open(filepath) as f:
        data = json.load(f)

    models = data["models"]
    print(f"Loading {len(models)} records...")

    conn = psycopg2.connect(**DB_CONN)
    cur = conn.cursor()
    inserted = skipped = 0

    for m in models:
        cur.execute("""
            INSERT INTO raw_bronze.config_model_dimensions (
                publisher_name, model_display_name, model_resource_name,
                model_family, model_variant, model_version,
                model_task, inference_scope, is_open_source,
                replicas, max_concurrency, ideal_concurrency, max_rps,
                accelerator_type, accelerators_per_replica, memory_gb,
                endpoint, tokens_per_second, avg_tokens_per_request,
                avg_latency_seconds, snapshot_date, source_file
            ) VALUES (
                %(publisher_name)s, %(model_display_name)s, %(model_resource_name)s,
                %(model_family)s, %(model_variant)s, %(model_version)s,
                %(model_task)s, %(inference_scope)s, %(is_open_source)s,
                %(replicas)s, %(max_concurrency)s, %(ideal_concurrency)s, %(max_rps)s,
                %(accelerator_type)s, %(accelerators_per_replica)s, %(memory_gb)s,
                %(endpoint)s, %(tokens_per_second)s, %(avg_tokens_per_request)s,
                %(avg_latency_seconds)s, %(snapshot_date)s, 'model_configuration.json'
            )
            ON CONFLICT (model_variant, snapshot_date) DO NOTHING
        """, m)
        if cur.rowcount == 1:
            inserted += 1
        else:
            skipped += 1

    conn.commit()
    cur.close()
    conn.close()
    print(f"Done — inserted: {inserted}, skipped (already exists): {skipped}")


def load_model_region_availability():
    """Load model_region_availability.json → raw_bronze.config_model_region_availability

    Note: routing_strategy and inference_region fields in the JSON are NOT stored
    in the bronze table — they will be derived in the silver layer.
    """

    filepath = os.path.join(CONFIG_DIR, "model_region_availability.json")
    with open(filepath) as f:
        data = json.load(f)

    records = data["model_region_availability"]
    print(f"Loading {len(records)} records...")

    conn = psycopg2.connect(**DB_CONN)
    cur = conn.cursor()
    inserted = skipped = 0

    for r in records:
        cur.execute("""
            INSERT INTO raw_bronze.config_model_region_availability (
                model_variant, source_region, deployed_at,
                is_active, snapshot_date, source_file
            ) VALUES (
                %(model_variant)s, %(source_region)s, %(deployed_at)s,
                %(is_active)s, %(snapshot_date)s, 'model_region_availability.json'
            )
            ON CONFLICT (model_variant, source_region, snapshot_date) DO NOTHING
        """, r)
        if cur.rowcount == 1:
            inserted += 1
        else:
            skipped += 1

    conn.commit()
    cur.close()
    conn.close()
    print(f"Done — inserted: {inserted}, skipped (already exists): {skipped}")


with DAG(
    dag_id="load_config_bronze",
    description="One-time load of model config JSON files into bronze tables",
    start_date=datetime(2026, 1, 1),
    schedule=None,    # manual trigger only — never runs on a schedule
    catchup=False,
    tags=["bronze", "config", "one-time"],
) as dag:

    task_models = PythonOperator(
        task_id="load_model_configuration",
        python_callable=load_model_configuration,
    )

    task_regions = PythonOperator(
        task_id="load_model_region_availability",
        python_callable=load_model_region_availability,
    )

    # Load model config first, then region availability
    task_models >> task_regions
Key things to understand:

schedule=None — never runs automatically. Only runs when you manually trigger it.

ON CONFLICT ... DO NOTHING — if the record already exists, the insert is silently skipped. Makes the DAG safe to run more than once.

routing_strategy and inference_region from the JSON are skipped intentionally — not columns in the bronze table, will be derived in silver.

/opt/airflow/database/config — inside the container, maps to your local database/config/ via the volume mount configured in Runbook 1.

  2. Find load_config_bronze in the DAG list
  3. Click the ▶ play button on the right → Trigger DAG
  4. Click the DAG name to open it, then click a task circle to view the logs
DAG not appearing in the list?Airflow scans for DAGs every 30 seconds. Wait and refresh. Still missing? Check logs: docker compose logs airflow-scheduler --tail=20 — look for a Python syntax error in the DAG file.

Troubleshooting

dbt debug says "could not connect to server" ▼

PostgreSQL is not running. Check: docker compose ps. If postgres shows Exit: docker compose start postgres

"No such file or directory: /opt/airflow/dbt" ▼

dbt volume mount missing. Both Airflow services need:

      - ./dbt:/opt/airflow/dbt

Add it, save, then: docker compose down && docker compose up -d

Airflow DAG not appearing in the UI ▼

Wait 30 seconds and refresh. Still missing?

docker compose logs airflow-scheduler --tail=20

Look for a Python syntax error in the DAG file. Fix it and wait 30 seconds.

DAG task fails with "No such file: /opt/airflow/database/config" ▼

The database volume mount is missing from docker-compose.yml. This should have been set up in Runbook 1. Open docker-compose.yml and confirm both airflow-webserver and airflow-scheduler have this line in their volumes section:

      - ./database:/opt/airflow/database

If missing, add it, save, then: docker compose down && docker compose up -d

profiles.yml accidentally committed to GitHub ▼
git rm --cached dbt/profiles.yml
git commit -m "remove profiles.yml - contains password"
git push

Confirm dbt/profiles.yml is in .gitignore. Since this is a local password only, no further action is needed — but in production, rotate the password immediately.

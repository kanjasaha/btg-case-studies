Who does what

RoleResponsibilitiesNever does
Developer
(you)		 Writes models, tests locally in personal schema, creates feature branches, opens PRs, responds to review Run --target prod manually. Touch the prod schemas directly.
Reviewer
(teammate)		 Reviews SQL logic, checks test coverage, approves or requests changes on PRs Approve code without reading it. Merge without CI passing.
GitHub Actions
(robot)		 Runs CI on every PR, builds dev on merge, triggers prod on merge to main Approve code. Make judgment calls. Skip tests.

(scheduler)		 Runs dbt build --target prod on a schedule and on merge to main Run in personal or dev. Be triggered by humans in normal operation.

(senior dev or lead)		 Opens the PR from dev to main. Reviews what is in the release. Decides timing. Deploy directly. Skip the PR process.

Git branches explained

This project uses three types of branches. Understanding each one is essential before writing a single command.

BranchPurposeWho creates itMerges into
main The production branch. What Airflow runs. Protected — nobody pushes directly to main. Created once when the repo was initialized Nothing — this is the destination
dev The shared integration branch. Feature branches merge here first for team visibility. Created once — lives permanently alongside main main (via PR, on release)
feature/* Your personal working branch. One branch per piece of work. Short-lived — deleted after merge. You create one every time you start new work dev (via PR)
Never push directly to main or dev. Always work on a feature branch. Open a Pull Request. Let CI run. Get a review. Then merge.

Direct pushes to main bypass every safety check — no CI, no review, no audit trail. Most teams configure GitHub to block direct pushes to main entirely (branch protection rules — covered in Step 1).

Set up GitHub Actions and branch protection

GitHub Actions is already built into every GitHub repository — nothing to install. You just need to create workflow files and configure branch protection.

Enable branch protection on main

  1. Open your repository on GitHub
  2. Go to Settings → Branches → Add branch protection rule
  3. Branch name pattern: main
  4. Check Require a pull request before merging
  5. Check Require status checks to pass before merging
  6. Check Do not allow bypassing the above settings
  7. Click Create
What does branch protection do? It makes it physically impossible to push directly to main. Anyone who tries gets a rejection from GitHub. Every change must go through a Pull Request — which means CI runs, a reviewer sees it, and there is a permanent record of what changed and when.

Create the GitHub Actions folder

cd ~/Documents/btg-case-studies
mkdir -p .github/workflows
touch .github/workflows/.gitkeep
Where do workflow files live? GitHub Actions looks for workflow files in .github/workflows/ at the root of your repository. Any .yml file there is automatically picked up — no registration or configuration needed.
2

Add GitHub Secrets

GitHub Actions needs your database credentials to run dbt. You never put passwords in code — they go in GitHub Secrets, which are encrypted and injected as environment variables at runtime.

  1. Open your repo on GitHub → Settings → Secrets and variables → Actions
  2. Click New repository secret for each of the following:
Secret nameValue
DBT_HOSTlocalhost
DBT_PORT5432
DBT_USERmds_user
DBT_PASSWORDmds_password
DBT_DBNAMEbtg_resource_utilization
For this tutorial only — localhost limitation. In this local tutorial, PostgreSQL runs on your Mac. GitHub Actions runs on GitHub's servers — it cannot reach your Mac's localhost. For the CI workflow to actually connect to your database, you would need a cloud database (like Supabase or RDS).

In this runbook you will still create and practice the full CI workflow. The database connection will fail in GitHub Actions (because it cannot reach your Mac), but you will learn to simulate CI locally using the exact same commands — which is what matters for your development practice.

When you move to a real cloud database, just update these secrets with the cloud credentials and CI will work end-to-end automatically.
3

Create the CI workflow

This workflow runs automatically on every Pull Request. It builds only the models you changed and runs all their tests.

code .github/workflows/dbt_ci.yml

Type this exactly, then save with ⌘ + S:

# .github/workflows/dbt_ci.yml
# Runs automatically on every Pull Request to dev or main.
# Builds only modified models and runs all their tests.
# Blocks the PR from merging if any test fails.

name: dbt CI

on:
  pull_request:
    branches:
      - dev
      - main
    paths:
      - 'dbt/models/**'
      - 'dbt/tests/**'
      - 'dbt/macros/**'
      - 'dbt/seeds/**'
      - 'dbt/snapshots/**'

env:
  DBT_HOST: ${{ secrets.DBT_HOST }}
  DBT_PORT: ${{ secrets.DBT_PORT }}
  DBT_USER: ${{ secrets.DBT_USER }}
  DBT_PASSWORD: ${{ secrets.DBT_PASSWORD }}
  DBT_DBNAME: ${{ secrets.DBT_DBNAME }}

jobs:
  dbt-ci:
    name: dbt CI — build modified models and run tests
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: "3.11"

      - name: Install dbt
        run: |
          pip install dbt-core dbt-postgres
          cd dbt && dbt deps

      - name: Download production manifest for Slim CI
        uses: actions/download-artifact@v4
        with:
          name: dbt-manifest
          path: ./prod-state/
        continue-on-error: true
        # First run will not have a manifest yet — that is fine.
        # Without it, dbt builds everything. With it, only modified models.

      - name: Run Slim CI — modified models only
        run: |
          cd dbt
          if [ -f "../prod-state/manifest.json" ]; then
            echo "Prod manifest found — running Slim CI"
            dbt build \
              --select state:modified+ \
              --defer \
              --state ../prod-state/ \
              --target ci \
              --profiles-dir .
          else
            echo "No prod manifest — running full build"
            dbt build \
              --target ci \
              --profiles-dir .
          fi

      - name: Save manifest for next CI run
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: dbt-manifest
          path: dbt/target/manifest.json
          retention-days: 7
What is Slim CI? The key command is dbt build --select state:modified+ --defer --state ./prod-state/.

state:modified+ — only build models whose SQL actually changed in this PR, plus all their downstream dependents.

--defer --state ./prod-state/ — for any upstream model you didn't change, read from the production table instead of rebuilding it.

Result: CI only builds 2-3 models instead of the entire project. It's fast, cheap, and still runs all tests on full prod-scale data via defer.
What tests does CI run? CI runs ALL tests on every model it builds — both tests that existed before your PR and any new tests you added in your PR. This catches two things:

— Did your change break something that was already working? (existing tests)
— Does your new logic actually pass the quality checks you defined? (new tests)

Create the production workflow

This workflow triggers when code merges into main. It runs the full dbt build against production schemas. Airflow also runs on a schedule — this workflow handles the code-change-triggered deployment.

code .github/workflows/dbt_prod.yml
# .github/workflows/dbt_prod.yml
# Triggers when code merges to main.
# Runs the full production build — all models, all tests.
# Nobody runs this manually — GitHub triggers it on merge.

name: dbt Production

on:
  push:
    branches:
      - main       # triggers on merge to main
  workflow_dispatch: # allows manual trigger from GitHub UI if needed

env:
  DBT_HOST: ${{ secrets.DBT_HOST }}
  DBT_PORT: ${{ secrets.DBT_PORT }}
  DBT_USER: ${{ secrets.DBT_USER }}
  DBT_PASSWORD: ${{ secrets.DBT_PASSWORD }}
  DBT_DBNAME: ${{ secrets.DBT_DBNAME }}

jobs:
  dbt-prod:
    name: dbt Production — full build and test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: "3.11"

      - name: Install dbt
        run: |
          pip install dbt-core dbt-postgres
          cd dbt && dbt deps

      - name: Check source freshness
        run: |
          cd dbt
          dbt source freshness --profiles-dir . --target prod
        continue-on-error: true

      - name: Run full production build
        run: |
          cd dbt
          dbt build --target prod --profiles-dir .

      - name: Save production manifest
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: dbt-manifest
          path: dbt/target/manifest.json
          retention-days: 7
          # This manifest is used by CI on the next PR for Slim CI comparison
Why does prod save the manifest? After each successful production run, GitHub saves the manifest.json as an artifact. The next time a developer opens a Pull Request, the CI workflow downloads this manifest and uses it to determine which models changed — enabling Slim CI. This is the link between prod and CI.

Commit both workflow files:

cd ~/Documents/btg-case-studies
git add .github/
git commit -m "add CI and production GitHub Actions workflows"
git push origin dev
On GitHub, under your repo's Actions tab, you see dbt CI and dbt Production listed as available workflows.

Create your first feature branch

From this point forward you never work directly on main or dev. Every piece of work starts with a feature branch.

The feature branch naming convention

FormatExample
feature/short-descriptionfeature/add-customer-revenue-ytd
fix/short-descriptionfix/null-tokens-in-stg-open-source
refactor/short-descriptionrefactor/simplify-int-revenue-daily

Start new work — always from dev

# Always start from the latest dev branch
cd ~/Documents/btg-case-studies
git checkout dev
git pull origin dev

# Create your feature branch
git checkout -b feature/add-weekly-token-model

# Confirm you are on the right branch
git branch
The asterisk (*) is next to feature/add-weekly-token-model. You are on your feature branch.

Work in your personal schema

Make your changes to dbt models. Run them in your personal schema as you develop:

# Build and test your specific model in your personal schema
# (--target personal is the default, so you can omit it)
dbt build --select token_usage_model_weekly

# Once happy, simulate CI locally before opening the PR
mkdir -p prod-state
cp dbt/target/manifest.json prod-state/manifest.json

dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod-state/ \
  --target personal
If this passes locally, CI will almost certainly pass too. CI runs the exact same command. Running it locally first saves you from waiting for GitHub Actions to tell you something you could have caught in 30 seconds on your laptop.

Commit your work

git add dbt/models/marts/token_usage_model_weekly.sql
git add dbt/models/marts/schema.yml
git commit -m "add weekly token usage mart model"
git push origin feature/add-weekly-token-model
What makes a good commit message? Short, specific, present tense. Says what changed:
✅ "add weekly token usage mart model"
✅ "fix null handling in stg_customer_details"
❌ "stuff"
❌ "fixed"
❌ "wip"
6

Open a Pull Request

A Pull Request (PR) is a formal proposal to merge your feature branch into another branch. It is where code review happens, CI runs, and the decision to merge is made.

Open the PR on GitHub

  1. Go to your repository on GitHub
  2. You will see a yellow banner: "feature/add-weekly-token-model had recent pushes" → click Compare & pull request
  3. Set the base branch to dev (not main — you are merging to dev first)
  4. Write a clear title: Add weekly token usage mart model
  5. In the description, explain: what you built, why, and how to verify it
  6. Click Create pull request
What makes a good PR description?
What: Added token_usage_model_weekly to mart_gold. Aggregates token usage per model per week from int_token_usage_minute.

Why: Business requested weekly granularity for the Metabase weekly review dashboard.

How to verify: Run dbt run --select token_usage_model_weekly and query SELECT * FROM mart_gold.token_usage_model_weekly LIMIT 10. Should return rows grouped by ISO week.

Tests added: not_null on all key columns, unique on (model_variant, week_start_date).
7

Watch CI run automatically

The moment you opened the PR, GitHub Actions started running the CI workflow. You did not do anything — it just happened.

What is happening right now

  1. GitHub detects the PR and triggers the dbt CI workflow
  2. A fresh Ubuntu machine spins up on GitHub's servers
  3. It installs Python, dbt, and your packages
  4. It downloads the latest production manifest (if one exists)
  5. It runs dbt build --select state:modified+ --defer --state ./prod-state/
  6. Results are posted back to your PR as a status check

Watch it in real time

On your PR page, scroll down to see the Checks section. Click Details next to the CI check to watch the live log output. You will see dbt building your models and running tests in real time.

The CI check shows a green ✅ and your PR shows "All checks have passed". The merge button is now active.
If CI fails — click Details to see the log. dbt will show you exactly which model failed and which test violated. Fix the issue in your local personal schema, commit the fix, and push. CI runs again automatically on the new commit.

Simulating CI locally when debugging

You don't have to wait for GitHub Actions to tell you something failed. Run CI locally:

dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod-state/ \
  --target personal

Same command, same result — but on your laptop in seconds instead of waiting for GitHub.

8

Merge to dev, then to main

Merge feature branch → dev

CI passed. A teammate reviewed and approved. Now merge:

  1. On the PR page, click Merge pull request → Confirm merge
  2. GitHub merges your feature branch into dev
  3. GitHub Actions detects the merge to dev and automatically runs the dev build
  4. Your model now exists in the dev schemas (dev_staging_silver, dev_mart_gold)
What happens in dev after the merge? The dev GitHub Actions workflow runs dbt build --target dev — a full build of all models into the dev schemas. This is the integration check: does your model work alongside everyone else's models that have already merged to dev? This is the step that catches conflicts between developers' work.

After merging, delete the feature branch — it has served its purpose:

git checkout dev
git pull origin dev
git branch -d feature/add-weekly-token-model

Release: merge dev → main

When the team is ready to release (could be daily, weekly, or after several features accumulate in dev), a release manager opens a PR from dev to main:

  1. Go to GitHub → Pull requests → New pull request
  2. Base: main, Compare: dev
  3. Title: Release — week of [date]
  4. Describe what is in this release (all features merged to dev since last release)
  5. CI runs one more time against the combined dev changes
  6. Senior developer reviews and approves
  7. Merge
What happens the moment you merge to main? The production GitHub Actions workflow triggers automatically. It runs dbt build --target prod — rebuilding all models in the prod schemas. Airflow will also pick up the changes on its next scheduled run if the GitHub Actions run is faster than the schedule.

From that point on, prod_mart_gold contains your new weekly token usage model. Metabase will see it immediately.
You just shipped a data model to production without touching a single prod command. You wrote the model, tested it locally, opened a PR, passed CI, got a review, and merged. Everything else happened automatically. This is how professional data teams work.

Team simulation — Alice and Bob working in parallel

The real power of this workflow becomes clear when multiple developers are working at the same time. Here is how Alice and Bob each add a model without stepping on each other.

👩 Alice — adding a new mart model
git checkout dev && git pull
git checkout -b feature/alice-customer-ytd

# Alice builds and tests in her personal schema
dbt build --select customer_revenue_ytd --target personal

# Alice simulates CI locally
dbt build --select state:modified+ --defer --state ./prod-state/

# Alice pushes and opens PR to dev
git push origin feature/alice-customer-ytd
# → Opens PR on GitHub
# → CI runs automatically
# → Teammate reviews
# → Merges to dev
👨 Bob — fixing a bug in staging at the same time
git checkout dev && git pull
git checkout -b fix/bob-null-tokens

# Bob builds and tests in HIS OWN personal schema
# Bob's schema: dbt_bob_staging_silver, dbt_bob_mart_gold
# Alice's work in dbt_alice_* is completely separate
dbt build --select stg_token_usage_open_source --target personal

# Bob simulates CI locally
dbt build --select state:modified+ --defer --state ./prod-state/

# Bob pushes and opens PR to dev
git push origin fix/bob-null-tokens
# → Opens PR on GitHub
# → CI runs automatically — only Bob's changed model is built
# → Alice's PR does not interfere with Bob's
# → Teammate reviews
# → Merges to dev
Why they never conflict. Alice's personal schema is dbt_alice_*. Bob's is dbt_bob_*. They write to completely separate schemas and never overwrite each other's work.

CI for Alice's PR only builds Alice's changed models. CI for Bob's PR only builds Bob's changed models. They run in parallel on GitHub with no interference.

When both PRs merge to dev, the full dev build runs and confirms their changes work together — that is the integration check that dev provides.

After both merge to dev

The release manager opens a PR from dev to main. The PR contains both Alice's new mart model and Bob's bug fix. CI runs once more on the combined changes. It passes. The PR merges. Production now has both changes — deployed automatically, with a full audit trail.

This is production-grade data engineering. Two developers worked independently, neither blocked the other, every change was reviewed and tested, and production was updated automatically. Nobody typed a prod command. Nobody broke anything.

Troubleshooting

CI fails with "could not connect to server" ▼

Expected in this tutorial — GitHub Actions cannot reach your Mac's PostgreSQL. This is the localhost limitation described in Step 2. The workflow itself is correct. To fix for real: migrate to a cloud database (Supabase free tier, or Neon) and update the GitHub Secrets with the cloud connection details.

To verify your CI command is correct locally: dbt build --select state:modified+ --defer --state ./prod-state/ --target personal

GitHub says "No changes to push" when I push my feature branch ▼

You have not committed anything yet. Run git status to see what files changed. Add them with git add dbt/models/path/to/your/model.sql and commit with git commit -m "your message". Then push.

dbt build --select state:modified+ shows "no nodes selected" ▼

dbt cannot detect any changes compared to the prod manifest. This means either: (1) the manifest in prod-state/ is from the same state as your current code, or (2) you haven't saved your model file yet. Try saving the file again and running dbt compile first to confirm the file is being picked up.

I accidentally committed to main directly ▼

If branch protection is enabled, GitHub will have rejected the push. If you somehow got through:

# See what commit you made
git log --oneline -3

# Move that commit to a feature branch instead
git checkout -b feature/accidental-main-commit
git push origin feature/accidental-main-commit

# Reset main to the previous state
git checkout main
git reset --hard HEAD~1
git push --force origin main

Enable branch protection immediately after to prevent this happening again.

My PR conflicts with dev — Git says "merge conflict" ▼

Another developer merged something to dev that touches the same file as your PR. Resolve it:

git checkout feature/your-branch
git fetch origin
git merge origin/dev

VS Code will highlight the conflicts. Edit the file to keep the correct version, then git add and git commit. Push your branch — the PR updates automatically.

How do I undo a merge that went wrong? ▼

If you merged to dev and something broke, the cleanest fix is a new PR that reverts the change — not a force reset (which rewrites history everyone else has already pulled):

git checkout dev
git pull origin dev
git revert HEAD    # creates a new commit that undoes the last merge
git push origin dev

This keeps the history clean and is safe for a shared branch.

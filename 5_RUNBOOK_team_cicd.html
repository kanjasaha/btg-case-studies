Git branches explained

This project uses three types of branches. Understanding each one is essential before writing a single command.

BranchPurposeWho creates itMerges into
main The production branch. What Airflow runs. Protected — nobody pushes directly to main. Created once when the repo was initialized Nothing — this is the destination
dev The shared integration branch. Feature branches merge here first for team visibility. Created once — lives permanently alongside main main (via PR, on release)
feature/* Your personal working branch. One branch per piece of work. Short-lived — deleted after merge. You create one every time you start new work dev (via PR)
Never push directly to main or dev. Always work on a feature branch. Open a Pull Request. Let CI run. Get a review. Then merge.

Direct pushes to main bypass every safety check — no CI, no review, no audit trail. Most teams configure GitHub to block direct pushes to main entirely (branch protection rules — covered in Step 1).

Set up the self-hosted runner

Why a self-hosted runner is required here. GitHub Actions has two types of runners:

Cloud runners (runs-on: ubuntu-latest) — GitHub spins up a fresh VM in the cloud. Fast and zero maintenance. But it cannot reach your Mac's PostgreSQL — your database lives inside Docker on your laptop with no public IP. The cloud runner has no way to connect to localhost:5432.

Self-hosted runner (runs-on: self-hosted) — a small process you run on your Mac. It registers with GitHub, listens for jobs, and executes them locally where it can reach localhost:5432 directly.

This is the standard solution for teams whose database is not publicly accessible.

Step 1 — Install GitHub CLI and verify authentication

We use the GitHub CLI (gh) to register the runner and set secrets from the terminal — no clicking in the UI required.

brew install gh
gh auth status
You see Logged in to github.com and your username. Token scopes must include repo and workflow. If not — run gh auth login and re-authenticate.

Step 2 — Get a runner registration token

gh api --method POST repos/YOUR-USERNAME/btg-case-studies/actions/runners/registration-token \
  --jq .token

Copy the token — it expires in one hour.

Replace YOUR-USERNAME with your actual GitHub username. For example: repos/kanjasaha/btg-case-studies/...

Step 3 — Download and configure the runner

# Create the runner folder
mkdir ~/actions-runner && cd ~/actions-runner

# Download — Apple Silicon (ARM64)
curl -o actions-runner-osx-arm64.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.322.0/actions-runner-osx-arm64-2.322.0.tar.gz

# Extract
tar xzf ./actions-runner-osx-arm64.tar.gz

# Configure — replace TOKEN with the token from Step 2
./config.sh --url https://github.com/YOUR-USERNAME/btg-case-studies --token TOKEN
Intel Mac? Replace arm64 with x64 in the download URL.

config.sh will ask three questions — press Enter to accept defaults for all:

Enter the name of the runner group: [press Enter for Default]
Enter the name of runner: [press Enter for mac]
Enter any additional labels: [press Enter to skip]

√ Runner successfully added
√ Runner connection is good

Enter name of work folder: [press Enter for _work]
√ Settings Saved.

Step 4 — Start the runner

Open a dedicated terminal tab and run:

cd ~/actions-runner
./run.sh
Leave this terminal open the entire time you are working. The runner only accepts jobs while ./run.sh is running. Every other command in this runbook must be run in a different terminal tab. If you close this tab or your Mac sleeps, the runner goes offline and GitHub Actions jobs will queue and wait.

You will see: Listening for Jobs — this confirms the runner is ready.
On GitHub → Settings → Actions → Runners, your Mac appears with a green dot and status Idle. In the terminal: Listening for Jobs.

Step 5 — Commit the CI-safe profiles.yml

The self-hosted runner runs dbt inside Docker using a profiles.yml from your repo. This is different from your personal profiles.yml — it contains no real passwords, only env_var() references that are filled in at runtime from GitHub Secrets.

Common mistake — profiles.yml blocked by .gitignore. The resource-utilization/.gitignore file blocks dbt/profiles.yml. The CI-safe version needs to be force-added to override this. If you skip this step, CI will fail with: Could not find profile named 'resource_utilization'

Confirm the file exists and force-add it:

# Confirm the file exists
ls resource-utilization/dbt/profiles.yml

# Force-add it (overrides the .gitignore rule)
git add -f resource-utilization/dbt/profiles.yml

# Verify it is staged
git status resource-utilization/dbt/profiles.yml
Git status shows new file: resource-utilization/dbt/profiles.yml under "Changes to be committed".
Why is it safe to commit this profiles.yml? Unlike your personal profiles.yml which has hardcoded passwords, this version uses env_var('DBT_USER', 'mds_user') — dbt reads the value from an environment variable at runtime. GitHub Secrets inject those variables. No real credentials are stored in the file itself.
Why does the runner use Docker to run dbt? The self-hosted runner runs on your Mac but dbt is not installed on your Mac directly — it is installed inside the Airflow Docker container. When the runner executes a workflow step, it runs in a plain non-interactive shell that does not load ~/.zshrc, so the dbt shell function you set up in Runbook 2 is not available.

The fix is to call dbt via docker compose run with the full path to the dbt binary inside the container: /usr/local/airflow/dbt_venv/bin/dbt. This is why the workflow files use docker compose commands instead of plain dbt commands.

Commit:

git commit -m "add CI-safe profiles.yml for GitHub Actions"
git push origin feature/test-ci

Set up GitHub Actions and branch protection

GitHub Actions is already built into every GitHub repository — nothing to install. You need to create the workflow folder and configure branch protection so no one can push directly to main.

Enable branch protection on main

Using the GitHub CLI — faster and repeatable than clicking through the UI:

gh api repos/YOUR-USERNAME/btg-case-studies/branches/main/protection \
  --method PUT \
  --header "Content-Type: application/json" \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["dbt CI — build modified models and run tests"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
What this protection does:

✅ PR required before merging to main — no direct pushes
✅ CI must pass before the merge button activates
✅ Even you (admin) must go through a PR
❌ No human reviewer required — correct for solo work

When you have a teammate, add the reviewer requirement by changing "required_pull_request_reviews": null to "required_pull_request_reviews": {"required_approving_review_count": 1}.
Use --input - with a heredoc, not --field. The --field flag sends strings. The branch protection API requires JSON objects. Using --field causes a 422 Invalid request error. The heredoc approach with --input - sends correctly structured JSON.
The API returns a JSON response with required_status_checks and enforce_admins — no error message. Main is now protected.
What does branch protection do? It makes it physically impossible to push directly to main. Anyone who tries gets a rejection from GitHub. Every change must go through a Pull Request — which means CI runs, a reviewer sees it, and there is a permanent record of what changed and when.

Create the GitHub Actions folder

cd ~/Documents/btg-case-studies
mkdir -p .github/workflows
touch .github/workflows/.gitkeep
Where do workflow files live? GitHub Actions looks for workflow files in .github/workflows/ at the root of your repository. Any .yml file there is automatically picked up — no registration needed.

Add GitHub Secrets

GitHub Actions needs your database credentials to run dbt. You never put passwords in code — they go in GitHub Secrets, encrypted and injected as environment variables at runtime.

Add all five secrets in one go using the GitHub CLI:

gh secret set DBT_HOST --body "localhost" --repo YOUR-USERNAME/btg-case-studies
gh secret set DBT_PORT --body "5432" --repo YOUR-USERNAME/btg-case-studies
gh secret set DBT_USER --body "mds_user" --repo YOUR-USERNAME/btg-case-studies
gh secret set DBT_PASSWORD --body "mds_password" --repo YOUR-USERNAME/btg-case-studies
gh secret set DBT_DBNAME --body "btg_resource_utilization" --repo YOUR-USERNAME/btg-case-studies
Secret nameValue for this local stackWhy
DBT_HOSTlocalhostSelf-hosted runner runs on your Mac — localhost hits Docker PostgreSQL directly
DBT_PORT5432PostgreSQL default port, exposed by Docker
DBT_USERmds_userThe PostgreSQL user created in Runbook 2
DBT_PASSWORDmds_passwordThe PostgreSQL password created in Runbook 2
DBT_DBNAMEbtg_resource_utilizationYour database name
Why localhost works with a self-hosted runner. Because the self-hosted runner is a process running on your Mac, localhost in any workflow step resolves to your Mac — exactly where Docker and PostgreSQL are running. No VPN, no tunnel, no public IP needed.
Moving to a cloud database later? Update these secrets with the cloud host and credentials. Switch to runs-on: ubuntu-latest — cloud databases have public endpoints that GitHub's cloud runners can reach. The workflow YAML stays identical; only the secrets change.

Create the CI workflow

This workflow runs automatically on every Pull Request. It builds only the models you changed and runs all their tests.

Critical: dbt commands must use Docker — not the shell function. The self-hosted runner runs in a plain non-interactive shell that does not load ~/.zshrc. The dbt shell function you set up in Runbook 2 is not available. If you use plain dbt deps or dbt build in the workflow, the runner will fail with:
line 1: dbt: command not found

The fix is to call dbt via docker compose run with the full binary path: /usr/local/airflow/dbt_venv/bin/dbt. This is why all dbt commands in the workflow use docker compose.
Note on paths. The repo root is btg-case-studies/ and the dbt project is inside resource-utilization/dbt/. All paths in the workflow are relative to the repo root where the runner checks out the code. 
code .github/workflows/dbt_ci.yml

Paste this exactly, then save with ⌘ + S:

# .github/workflows/dbt_ci.yml
# Runs automatically on every Pull Request to dev or main.
# Builds only modified models and runs all their tests.
# Blocks the PR from merging if any test fails.
# Runs on your Mac via the self-hosted runner — required to reach localhost PostgreSQL.

name: dbt CI

on:
  pull_request:
    branches:
      - dev
      - main
    paths:
      - 'resource-utilization/dbt/models/**'
      - 'resource-utilization/dbt/tests/**'
      - 'resource-utilization/dbt/macros/**'
      - 'resource-utilization/dbt/seeds/**'
      - 'resource-utilization/dbt/snapshots/**'

env:
  DBT_HOST: ${{ secrets.DBT_HOST }}
  DBT_PORT: ${{ secrets.DBT_PORT }}
  DBT_USER: ${{ secrets.DBT_USER }}
  DBT_PASSWORD: ${{ secrets.DBT_PASSWORD }}
  DBT_DBNAME: ${{ secrets.DBT_DBNAME }}

jobs:
  dbt-ci:
    name: dbt CI — build modified models and run tests
    runs-on: self-hosted

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install dbt packages
        run: |
          docker compose -f resource-utilization/docker-compose.yml \
            run --rm airflow-scheduler bash -c \
            "cd /opt/airflow/dbt && /usr/local/airflow/dbt_venv/bin/dbt deps"

      - name: Download production manifest for Slim CI
        uses: actions/download-artifact@v4
        with:
          name: dbt-manifest
          path: ./prod-state/
        continue-on-error: true
        # First run will not have a manifest — that is fine.
        # Without it, dbt builds everything. With it, only modified models.

      - name: Run Slim CI — modified models only
        run: |
          if [ -f "prod-state/manifest.json" ]; then
            echo "Prod manifest found — running Slim CI"
            docker compose -f resource-utilization/docker-compose.yml \
              run --rm airflow-scheduler bash -c \
              "cd /opt/airflow/dbt && /usr/local/airflow/dbt_venv/bin/dbt build \
              --select state:modified+ \
              --defer \
              --state /opt/airflow/prod-state/ \
              --target ci \
              --profiles-dir ."
          else
            echo "No prod manifest — running full build"
            docker compose -f resource-utilization/docker-compose.yml \
              run --rm airflow-scheduler bash -c \
              "cd /opt/airflow/dbt && /usr/local/airflow/dbt_venv/bin/dbt build \
              --target ci \
              --profiles-dir ."
          fi

      - name: Save manifest for next CI run
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: dbt-manifest
          path: resource-utilization/dbt/target/manifest.json
          retention-days: 7
What is Slim CI? The key command is dbt build --select state:modified+ --defer --state ./prod-state/.

state:modified+ — only build models whose SQL changed in this PR, plus their downstream dependents.

--defer --state ./prod-state/ — for any upstream model you didn't change, read from prod instead of rebuilding it. Output lands in the dbt_ci schema — not in prod.

Result: CI only builds 2–3 models instead of the entire project. Fast, and still tests against real prod-scale data via defer.

Create the production workflow

This workflow triggers when code merges into main. It runs the full dbt build against production schemas.

code .github/workflows/dbt_prod.yml

Paste this exactly, then save with ⌘ + S:

# .github/workflows/dbt_prod.yml
# Triggers when code merges to main.
# Runs the full production build — all models, all tests.
# Nobody runs this manually — GitHub triggers it on merge.
# Uses Docker to call dbt — same reason as dbt_ci.yml.

name: dbt Production

on:
  push:
    branches:
      - main
  workflow_dispatch:

env:
  DBT_HOST: ${{ secrets.DBT_HOST }}
  DBT_PORT: ${{ secrets.DBT_PORT }}
  DBT_USER: ${{ secrets.DBT_USER }}
  DBT_PASSWORD: ${{ secrets.DBT_PASSWORD }}
  DBT_DBNAME: ${{ secrets.DBT_DBNAME }}

jobs:
  dbt-prod:
    name: dbt Production — full build and test
    runs-on: self-hosted

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install dbt packages
        run: |
          docker compose -f resource-utilization/docker-compose.yml \
            run --rm airflow-scheduler bash -c \
            "cd /opt/airflow/dbt && /usr/local/airflow/dbt_venv/bin/dbt deps"

      - name: Run full production build
        run: |
          docker compose -f resource-utilization/docker-compose.yml \
            run --rm airflow-scheduler bash -c \
            "cd /opt/airflow/dbt && /usr/local/airflow/dbt_venv/bin/dbt build \
            --target prod \
            --profiles-dir ."

      - name: Save production manifest
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: dbt-manifest
          path: resource-utilization/dbt/target/manifest.json
          retention-days: 7
Why does prod save the manifest? After each successful production run, GitHub saves manifest.json as an artifact. The next time a developer opens a PR, the CI workflow downloads it to determine which models changed — enabling Slim CI. This is the link between prod and CI.

Watch CI run automatically

The moment you opened the PR, GitHub Actions started the CI workflow. You did not do anything — it just happened.

What is happening right now

  1. GitHub detects the PR and sends the job to your self-hosted runner
  2. The runner on your Mac checks out the code
  3. It runs dbt deps via Docker to install packages
  4. It downloads the latest production manifest (if one exists)
  5. It runs dbt build --select state:modified+ via Docker
  6. Results are posted back to your PR as a status check
Why your Mac and not GitHub's servers? The self-hosted runner runs on your Mac — so it can reach localhost:5432 directly. GitHub's cloud runners are on remote servers that cannot reach your local PostgreSQL.

Monitor from the terminal

# Check the status of the latest run
gh run list --repo YOUR-USERNAME/btg-case-studies --limit 1

# Check all checks on your PR
gh pr checks 1 --repo YOUR-USERNAME/btg-case-studies
PR checks show: All checks were successful — 1 successful. The runner terminal shows: Job completed with result: Succeeded.

If CI fails — get the logs

# See the failed log output
gh run view --repo YOUR-USERNAME/btg-case-studies --log-failed
Common CI failures and fixes:

dbt: command not found — The workflow is using plain dbt instead of docker compose run. The self-hosted runner's shell doesn't load ~/.zshrc so the dbt shell function isn't available. Fix: use the full Docker command as shown in Step 3.

Could not find profile named 'resource_utilization' — The profiles.yml was not committed. It's blocked by resource-utilization/.gitignore. Fix: git add -f resource-utilization/dbt/profiles.yml and push.

POSTGRES_USER variable is not set — Docker is reading from .env which is gitignored and not present in the runner's working directory. These warnings are harmless — the actual credentials come from GitHub Secrets injected as environment variables, not from .env.

Simulating CI locally before pushing

Run the exact same command CI runs — catch failures before they hit GitHub:

cd resource-utilization/dbt

  --state ../../prod-state/ \

Merge to dev, then to main

Merge feature branch → dev

CI passed. Now merge the PR using the GitHub CLI:

gh pr merge 1 \
  --repo YOUR-USERNAME/btg-case-studies \
  --squash \
  --delete-branch
What is squash merge? --squash combines all commits from your feature branch into a single clean commit on dev. Instead of seeing every "WIP" and "fix typo" commit in the dev history, you get one meaningful commit per feature. This keeps the git log readable.
What happens in dev after the merge? The dev GitHub Actions workflow triggers and runs dbt build --target dev — a full build of all models into the dev schemas. This is the integration check: does your model work alongside everyone else's models? This catches conflicts between developers' work before anything reaches production.

Release: merge dev → main using the GitHub CLI

When the team is ready to release, open a PR from dev to main:

gh pr create \
  --repo YOUR-USERNAME/btg-case-studies \
  --base main \
  --head dev \
  --title "Release — $(date +%Y-%m-%d)" \
  --body "Full release — all features merged to dev since last release"

CI runs one more time. Once it passes and a reviewer approves, merge:

gh pr merge 2 \
  --repo YOUR-USERNAME/btg-case-studies \
  --squash
What happens the moment you merge to main? The production GitHub Actions workflow triggers automatically. It runs dbt build --target prod — rebuilding all models in the prod schemas. Airflow will also pick up the changes on its next scheduled run.

From that point, prod_mart_gold has your new model. Metabase sees it immediately.
You just shipped a data model to production without touching a single prod command. You wrote the model, tested locally, opened a PR, passed CI, got a review, and merged. Everything else happened automatically.

Troubleshooting

CI fails with "could not connect to server" ▼

The self-hosted runner is running but cannot reach PostgreSQL. Check:

  1. Docker is running — docker compose ps should show all 5 containers Up
  2. The runner is active — your ./run.sh terminal must show Listening for Jobs
  3. The credentials in GitHub Secrets match your database — DBT_USER=mds_user, DBT_PASSWORD=mds_password
  4. The profiles.yml in the repo uses host: localhost for all targets (not postgres — that's only for inside Docker)

Verify locally: dbt build --select state:modified+ --defer --state ./prod-state/ --target ci --profiles-dir dbt/

GitHub says "No changes to push" when I push my feature branch ▼

You have not committed anything yet. Run git status to see what files changed. Add them with git add dbt/models/path/to/your/model.sql and commit with git commit -m "your message". Then push.

dbt build --select state:modified+ shows "no nodes selected" ▼

dbt cannot detect any changes compared to the prod manifest. This means either: (1) the manifest in prod-state/ is from the same state as your current code, or (2) you haven't saved your model file yet. Try saving the file again and running dbt compile first to confirm the file is being picked up.

Push rejected — "Updates were rejected because a pushed branch tip is behind its remote" ▼

You are trying to push to a branch that has commits on GitHub that you don't have locally. This happens when you make a commit on the wrong branch and then try to push. Fix it:

# Step 1 — set git pull strategy (do this once globally)
git config --global pull.rebase false

# Step 2 — make sure you are on the right branch
git branch   # asterisk shows current branch

# Step 3 — if you committed on the wrong branch, move the commit
git checkout -b feature/rescue-branch   # create a new branch with your commit
git push origin feature/rescue-branch

# Step 4 — reset the wrong branch back to what's on GitHub
git checkout main
git fetch origin
git reset --hard origin/main

# Step 5 — now push your fix from the correct branch
git checkout feature/rescue-branch
git push origin feature/rescue-branch
git pull says "Need to specify how to reconcile divergent branches" ▼

Git doesn't know whether to merge or rebase when pulling. Set the default once and it never asks again:

git config --global pull.rebase false
git pull origin main
PR not mergeable — "base branch policy prohibits the merge" ▼

This usually means CI hasn't passed yet. Wait for the CI check to complete — the merge button activates automatically once it goes green.

If CI passed but the merge is still blocked, check that your branch protection is set to "required_pull_request_reviews": null — the solo-friendly setup requires CI but no human reviewer. Verify with:

gh api repos/YOUR-USERNAME/btg-case-studies/branches/main/protection \
  --jq '.required_pull_request_reviews'

Should return null. If it returns a JSON object with required_approving_review_count, re-run the branch protection setup command from Step 1 with "required_pull_request_reviews": null.

I accidentally committed to main directly ▼

If branch protection is enabled, GitHub will have rejected the push. If you somehow got through:

# See what commit you made
git log --oneline -3

# Move that commit to a feature branch instead
git checkout -b feature/accidental-main-commit
git push origin feature/accidental-main-commit

# Reset main to the previous state
git checkout main
git reset --hard HEAD~1
git push --force origin main

Enable branch protection immediately after to prevent this happening again.

My PR conflicts with dev — Git says "merge conflict" ▼

Another developer merged something to dev that touches the same file as your PR. Resolve it:

git checkout feature/your-branch
git fetch origin
git merge origin/dev

VS Code will highlight the conflicts. Edit the file to keep the correct version, then git add and git commit. Push your branch — the PR updates automatically.

How do I undo a merge that went wrong? ▼

If you merged to dev and something broke, the cleanest fix is a new PR that reverts the change — not a force reset (which rewrites history everyone else has already pulled):

git checkout dev
git pull origin dev
git revert HEAD    # creates a new commit that undoes the last merge
git push origin dev

This keeps the history clean and is safe for a shared branch.

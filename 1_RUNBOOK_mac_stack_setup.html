Decisions to make before you start

Before touching a single command, there are two things you need to decide. These are hard to change later — especially the database name.

Decide your database name now — before you start. The database name is set when Docker first starts PostgreSQL. It gets baked into the Airflow connection string, the Metabase connection, the dbt profile, and your Git history. Changing it later means updating all of those places and recreating the database from scratch.

In this runbook we use: btg_resource_utilization

Rules for naming:
Use underscores not hyphens — btg_resource_utilization ✅ not btg-resource-utilization ❌
Hyphens require double quotes every time you reference the database in SQL. Underscores never do.
Decide your project folder name now. In this runbook:
~/Documents/btg-case-studies/ — the Git repository (holds all case studies)
~/Documents/btg-case-studies/resource-utilization/ — this specific case study

If you use different names, substitute them everywhere you see these paths in this runbook.

Create your environment file

What is a .env file?A .env file holds private settings — passwords, port numbers — that your tools need to talk to each other. It lives only on your Mac and is blocked by .gitignore so it never goes to GitHub.
What is a .env.example file?A safe template with the same variable names but no real passwords. Anyone who clones the repo copies this to .env and fills in their own values. This one gets committed to Git.
Stop — confirm you are in the right folder.

You must see .../btg-case-studies/resource-utilization — not the parent folder. If you create .env in the wrong folder, nothing will work.

Already created it in the wrong folder? Fix it: 
cd ~/Documents/btg-case-studies
ls -a
If you see .env or .env.example here, delete them: 
rm .env .env.example
git add .
git commit -m "remove env files from wrong folder"
Then go to the right folder and continue.

Create the example template:

code .env.example

Type this exactly, then save with ⌘ + S:

# PostgreSQL — your data warehouse
POSTGRES_USER=mds_user
POSTGRES_PASSWORD=mds_password
POSTGRES_DB=btg_resource_utilization
POSTGRES_PORT=5432

# Airflow — your pipeline scheduler
AIRFLOW_UID=50000
AIRFLOW__WEBSERVER__SECRET_KEY=changeme123
What does each variable mean? POSTGRES_USER — the username dbt, Airflow, and Metabase use to connect to PostgreSQL. mds_user stands for Modern Data Stack user.

POSTGRES_PASSWORD — the password for that user.

POSTGRES_DB — the name of the database PostgreSQL creates on first start. This is hard to change later — choose carefully. We use btg_resource_utilization with underscores (not hyphens — see the Decisions section).

POSTGRES_PORT — port 5432 is the PostgreSQL standard default.

AIRFLOW_UID — tells Airflow which system user to run as inside Docker. Prevents permission errors on your Mac.

AIRFLOW__WEBSERVER__SECRET_KEY — a private key Airflow uses to secure browser sessions. Any random string works for local development.

Now copy it to create the real .env:

cp .env.example .env
What does cp do?cp = copy. This makes an exact copy of .env.example named .env. You now have two files — the safe template (committed to Git) and the real working file (blocked by .gitignore).

Commit the template — but not the real .env:

cd ~/Documents/btg-case-studies
git add .
git commit -m "add environment variable template"
git push
Git confirms the commit. Run git status — .env should not appear anywhere.

Create the Dockerfile

What is a Dockerfile? A Dockerfile is a recipe for building a custom Docker image. The official Airflow image comes with Airflow but nothing else. We need dbt and Cosmos installed inside it too.

Why is dbt inside Airflow instead of its own container?
The official dbt Docker image does not support Apple Silicon (M1/M2/M3) chips. Rather than a workaround, we install dbt directly inside the Airflow container using pip — which builds natively for your chip. This is also the correct pattern for Cosmos: dbt needs to live where Airflow lives. 
cd ~/Documents/btg-case-studies/resource-utilization
code Dockerfile

Type this exactly, then save with ⌘ + S:

FROM apache/airflow:2.11.2

# Install dbt in its own virtual environment
# Keeps dbt and Airflow dependencies separate — no version conflicts
USER root
RUN python -m venv /usr/local/airflow/dbt_venv && \
    /usr/local/airflow/dbt_venv/bin/pip install --no-cache-dir \
    dbt-core \
    dbt-postgres

# Switch back to airflow user — never run production software as root
USER airflow

# Install Airflow Python packages from requirements.txt
COPY airflow/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
What does each line mean? FROM apache/airflow:2.11.2 — start with the official Airflow image as the base.

USER root — switch to admin to install software. Like getting the manager's keys.

python -m venv /usr/local/airflow/dbt_venv — create a private Python environment just for dbt. Keeps dbt and Airflow from conflicting over package versions.

pip install dbt-core dbt-postgres — install dbt and its PostgreSQL adapter inside that isolated environment.

USER airflow — switch back to normal user for security.

COPY airflow/requirements.txt . — copy our package list into the image.

pip install -r requirements.txt — install everything listed (Cosmos and the Postgres provider).

Create docker-compose.yml

What is a .yml file?YAML (pronounced "yam-ul") is a way to write structured settings in plain English. Three rules: indentation = structure (2 spaces per level), spaces only never tabs, and most lines follow key: value.
What is docker-compose.yml?The blueprint for your entire stack. It tells Docker which services to run, how to configure each one, and how they connect to each other.
Why does Airflow run as three separate services? When you run docker compose ps you will see three Airflow-related containers. Here is what each one does:

airflow-init — a setup job, not a service. Runs once when you first start the stack. It migrates the Airflow database (creates all the tables Airflow needs to track DAGs, runs, and task history) and creates the admin user. Once finished it exits. You will never see it running again after the first start — that is normal and expected.

airflow-webserver — the UI. Serves the Airflow interface at localhost:8080. This is what you open in your browser to see your DAGs, trigger them manually, and read task logs.

airflow-scheduler — the engine. Runs silently in the background, watching your DAGs folder every 30 seconds for new or changed files, and triggering tasks when their schedule says it is time. No UI — just the worker. This is also the container we run dbt commands through, because dbt is installed inside it.

Why three instead of one? Separating them means each can be restarted independently. If the webserver crashes, the scheduler keeps running and your pipelines keep executing. This is how Airflow is deployed in production too — often across separate machines.
Container names must be unique across your entire Docker environment.If you run multiple stacks on the same Mac, containers from different projects can conflict. We prefix ours with btg- to prevent this. If you ever see a "container name already in use" error, it means another project has a container with the same name — use docker rm container-name to remove the old one first.
Airflow environment variables are hardcoded — not read from .env.Variables with double underscores like AIRFLOW__CORE__EXECUTOR conflict with how Docker Compose reads .env files. Docker strips the double underscores and can't find the variable, defaulting to blank. The fix is to write these values directly in the compose file.
code docker-compose.yml

Type this exactly, then save with ⌘ + S:

services:

  # ── PostgreSQL ──────────────────────────────────────────────
  # Your data warehouse. Stores all project data.
  postgres:
    image: postgres:17
    container_name: postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    ports:
      - "${POSTGRES_PORT}:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  # ── Airflow Init ─────────────────────────────────────────────
  # Runs once on startup: sets up the database and creates admin user
  # Exits automatically when done — this is expected behaviour
  airflow-init:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: btg-airflow-init
    depends_on:
      postgres:
        condition: service_started
    environment:
      AIRFLOW__CORE__EXECUTOR: LocalExecutor
      AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: postgresql+psycopg2://mds_user:mds_password@postgres:5432/btg_resource_utilization
      AIRFLOW__WEBSERVER__SECRET_KEY: changeme123
      _AIRFLOW_DB_MIGRATE: 'true'
      _AIRFLOW_WWW_USER_CREATE: 'true'
      _AIRFLOW_WWW_USER_USERNAME: admin
      _AIRFLOW_WWW_USER_PASSWORD: admin
    command: version
    restart: on-failure

  # ── Airflow Webserver ─────────────────────────────────────────
  # Serves the Airflow UI at localhost:8080
  airflow-webserver:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: btg-airflow-webserver
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_started
    environment:
      AIRFLOW__CORE__EXECUTOR: LocalExecutor
      AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: postgresql+psycopg2://mds_user:mds_password@postgres:5432/btg_resource_utilization
      AIRFLOW__WEBSERVER__SECRET_KEY: changeme123
    ports:
      - "8080:8080"
    volumes:
      - ./airflow/dags:/opt/airflow/dags
      - ./airflow/logs:/opt/airflow/logs
      - ./airflow/plugins:/opt/airflow/plugins
      - ./dbt:/opt/airflow/dbt
      - ./database:/opt/airflow/database    # config files and SQL scripts
    command: webserver

  # ── Airflow Scheduler ─────────────────────────────────────────
  # Watches for DAGs to trigger and runs tasks
  airflow-scheduler:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: btg-airflow-scheduler
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_started
    environment:
      AIRFLOW__CORE__EXECUTOR: LocalExecutor
      AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: postgresql+psycopg2://mds_user:mds_password@postgres:5432/btg_resource_utilization
      AIRFLOW__WEBSERVER__SECRET_KEY: changeme123
    volumes:
      - ./airflow/dags:/opt/airflow/dags
      - ./airflow/logs:/opt/airflow/logs
      - ./airflow/plugins:/opt/airflow/plugins
      - ./dbt:/opt/airflow/dbt
      - ./database:/opt/airflow/database    # config files and SQL scripts
    command: scheduler

  # ── Metabase ──────────────────────────────────────────────────
  # BI tool — turns your data into charts and dashboards
  # Needs its own database schema (we create it in Step 13)
  metabase:
    image: metabase/metabase:v0.59.2
    container_name: metabase
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_started
    ports:
      - "3000:3000"
    environment:
      MB_DB_TYPE: postgres
      MB_DB_DBNAME: metabase
      MB_DB_PORT: 5432
      MB_DB_USER: ${POSTGRES_USER}
      MB_DB_PASS: ${POSTGRES_PASSWORD}
      MB_DB_HOST: postgres
    volumes:
      - metabase_data:/metabase-data

  # ── pgAdmin ───────────────────────────────────────────────────
  # Visual interface for browsing your PostgreSQL database
  pgadmin:
    image: dpage/pgadmin4:latest
    container_name: btg-pgadmin
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_started
    environment:
      PGADMIN_DEFAULT_EMAIL: "[email protected]"    # quotes required — @ confuses YAML parser
      PGADMIN_DEFAULT_PASSWORD: "admin"
    ports:
      - "5050:80"
    volumes:
      - pgadmin_data:/var/lib/pgadmin

volumes:
  postgres_data:    # stores PostgreSQL data across restarts
  metabase_data:    # stores Metabase settings and dashboards
  pgadmin_data:     # stores pgAdmin server registrations
Email addresses in YAML must be quoted.The @ symbol confuses the YAML parser and causes a validation error when running docker compose build. Any value containing @ must be wrapped in double quotes. You will see this in the pgAdmin environment block — it is intentional.
What does depends_on: condition: service_started mean?Tells Docker: do not start this service until the named service has started first. Airflow needs PostgreSQL running first because it stores its own history in the database. The full format is required — the old shorthand (- postgres) throws a validation error in current Docker Compose versions.
What does ./dbt:/opt/airflow/dbt mean?This is a volume mount — it links your local dbt/ folder to a path inside the container. Whatever you create in dbt/ on your Mac is immediately visible inside the Airflow container at /opt/airflow/dbt. dbt needs this volume mount in both the webserver and scheduler services so Cosmos can find your models.

Create the Metabase database

Metabase stores its settings in a separate PostgreSQL database called metabase. PostgreSQL does not create this automatically — we create it manually once:

docker exec postgres psql -U mds_user -d btg_resource_utilization -c "CREATE DATABASE metabase;"
What does this command do? docker exec postgres — run a command inside the running postgres container.
psql -U mds_user -d btg_resource_utilization — connect using our credentials.
-c "CREATE DATABASE metabase;" — execute this SQL: create an empty database called metabase.

Restart Metabase so it picks up the new database:

docker compose restart metabase
After 1–2 minutes, docker compose ps shows metabase as Up.

Pushing future changes to GitHub

Your GitHub repo was created in Step 7. Every time you finish a piece of work, save it to GitHub with these three commands:

git add .
git commit -m "describe what changed"
git push
commit vs push — what is the difference?git commit saves a snapshot locally — like saving a draft. git push uploads it to GitHub — like publishing it. You can make many commits before pushing. The commits are still safe locally even if you haven't pushed yet.
What makes a good commit message?Short, specific, and in the present tense. Describe what changed and why:
✅ "add bronze table definitions"
✅ "fix profiles.yml database name"
❌ "stuff"
❌ "fixed it"

A good commit history reads like a story of exactly what you built and when.

Stopping and starting the stack

Stop everything (data is saved):

docker compose stop

Start again later:

docker compose start

Wipe everything and start completely fresh ⚠️ (deletes all data):

docker compose down -v
If you run multiple stacks on the same Mac, only one can use each port at a time. Ports 8080, 3000, 5432, and 5050 must be free before starting this stack. Stop other stacks first: go to their folder and run docker compose stop.

Troubleshooting

Click a problem to see the fix.

A container shows "Exit" instead of "Up" ▼

Check the logs to see what went wrong:

docker compose logs airflow-webserver --tail=30

Replace airflow-webserver with the service that failed.

"port is already in use" ▼

Another project is using that port. Stop it first, then retry:

docker compose down
docker compose up -d
"container name already in use" ▼

A leftover container from another project has the same name. Remove it:

docker rm container-name-from-error

Then retry docker compose up -d.

"command not found: docker" ▼

Docker isn't running. Open Applications → Docker. Wait for the 🐳 whale in your menu bar to stop animating, then try again.

Metabase stuck on loading screen ▼

Metabase takes 1–2 minutes on first start. Wait and refresh. Still stuck after 5 minutes?

Understanding your place in the pipeline

Before moving to Runbook 2, take two minutes to understand where this local setup fits in the bigger picture. This will make every decision in the next four runbooks make sense.

You just built your personal environment. Everything running on your Mac right now — PostgreSQL, Airflow, Metabase — is your personal sandbox. Nobody else can see it. You can break things, rebuild them, and experiment freely without affecting anyone else.

The four environments

In a real data engineering team, code travels through four environments before reaching end users. Each one serves a different purpose:

EnvironmentWhere it runsWho controls itPurpose
Personal 👈 you are here Your Mac (Docker) You alone Write and test your models. Break things freely. Nobody sees this.
Dev Shared database schemas The whole team Check that your work plays nicely with your teammates' work. Full bronze data, full project build.
CI (Continuous Integration) Automated — GitHub Actions Nobody — it's automatic Robot runs your tests every time you open a Pull Request. Blocks bad code from reaching production.
Production Shared database schemas Automated — nobody manually deploys Live, trusted data. Business users and dashboards read from here. Only clean, tested code reaches here.
The golden rule: Humans push code to branches. Automation pushes to production. You will never type a command that deploys to production — GitHub Actions does that for you automatically when your Pull Request is approved and merged.

How this all fits in one PostgreSQL instance

Because this is a local tutorial stack, all four environments live inside your single PostgreSQL database as separate schemas. In a real company they would be separate databases or even separate cloud accounts — but the concept is identical.

btg_resource_utilization (one PostgreSQL database) ├── raw_bronze ← shared source — all environments read from here │ ├── dbt_yourname_staging_silver ← YOUR personal schema (replace yourname with your name) ├── dbt_yourname_mart_gold ← YOUR personal schema │ ├── dev_staging_silver ← shared dev schema (team integration) ├── dev_mart_gold ← shared dev schema │ ├── prod_staging_silver ← production schema (Airflow writes here) └── prod_mart_gold ← production schema (Metabase reads from here)
Replace yourname with your actual name throughout these runbooks. For example if your name is Alice: dbt_alice_staging_silver, dbt_alice_mart_gold.
Use lowercase, no spaces. This is how teammates tell each other's schemas apart.
